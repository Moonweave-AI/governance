# 安全、隐私与责任边界

> 本文定义辉夜计划在软件、AI Agent、数据、模型、具身终端、基础设施与开放社区中的安全、隐私、合规与伦理边界。它不是完整的实施手册，而是所有工程标准、研发流程、发布门禁和事故响应的上位约束——是 `01-Principles.md` 里相关原则的硬底座。

安全不是发布前的最后检查，而是项目从立项、设计、实现、评测、发布、运行到退役全过程的基础约束。原则可以解释，工程规范可以迭代，但安全、隐私、合规、具身失效安全与未授权资产边界，不能被普通工程效率覆盖。

这份文档回答的问题：

| 问题                       | 对应章节             |
| -------------------------- | ----------------- |
| 什么东西可以进入项目？       | §4 资产与来源安全     |
| 什么项目必须被安全审查？     | §3 风险分级模型       |
| 什么时候不能继续推进？       | §7 Stop-Ship 条件   |
| AI Agent 能做什么、不能做什么？ | §8 AI / Agent 安全  |
| 具身终端何时能获得执行权？   | §9 具身智能与物理安全   |
| 用户数据、记忆、日志如何处理？ | §5 数据、隐私与记忆安全 |
| 事故发生后如何处理？         | §11 漏洞披露与事故响应 |
| 哪些行为即使技术可行也不做？ | §12 伦理边界与禁止项   |

整体框架吸收 [NIST CSF 2.0](https://csrc.nist.gov/pubs/cswp/29/the-nist-cybersecurity-framework-csf-20/final) 的风险治理思路：用统一分类法理解、评估、优先级排序和沟通安全工作，而不是直接规定某种实现方式。

## 1. 目的与适用范围

本文适用于辉夜计划下的全部产出：源代码、第三方依赖、二进制与容器镜像、数据集、模型权重与微调件、多媒体资产、Prompt 与评测集、具身资产、运行日志与长期记忆，以及承载这些的基础设施与开放社区。

它对三类对象设约束：

- **资产**：进入系统的一切东西，必须来源可信、许可清楚、用途明确（§4）。
- **变更**：项目从想法到生产的每一次演进，必须按风险分级通过对应门禁（§6）。
- **行为**：系统在运行中能做什么、不能做什么，必须服从权限与伦理边界（§8、§9、§12）。

本文不代替具体工程标准、RFC 或负责人拍板，但在任何冲突中优先级高于工程效率。具体做法（扫描工具选型、门禁脚本实现、许可证清单）下沉到 `04-Engineering` 各标准与本文件引用的规范；本文只定**哪些风险不可接受、哪些项目必须被审查、哪些证据必须留下、哪些发布必须阻断**。

## 2. 核心安全原则

以下八条是本项目的核心安全规则：

1. **来源可信** — 没有来源、许可、哈希、Owner 与用途边界的资产，不得进入核心仓库或生产系统。对齐 *资产清洁与来源可追溯*。
2. **权限最小** — Agent 的能力不等于权限；任何工具调用、数据写入、外部调用或物理动作，必须显式授权。
3. **行为可审计** — 关键决策必须留下可追踪、可审计、可在合理条件下重放的证据。对齐 *全局可观测与行为可追踪*。
4. **数据最小化** — 只收集完成目标所需的数据；交互数据不得默认用于训练。
5. **高风险默认受限** — 不确定时按可监督、可逆、可中止的程度逐步放开，优先选波及更小、更易收回的一步。
6. **物理执行权单独授权** — 物理执行权不是模型能力的自然延伸，而是需要被单独授予、验证和回收的高风险权限。对齐 *具身收敛与失效安全*。
7. **事故可恢复** — 不保证不出事故，但保证事故可预见、可隔离、可发现、可回滚、可复盘。
8. **例外必须过期** — 任何偏离安全要求的例外都必须有 Owner、缓解措施、过期时间与复审日期，不得突破法律、隐私、物理安全与未授权 IP 底线。

## 3. 风险分级模型

没有风险分级，后面所有要求都会变成"所有项目都要最高标准"，最终没人执行。

本项目采用 **S0–S5 + Blocked** 内部安全等级，风险分层思路对齐 [EU AI Act](https://digital-strategy.ec.europa.eu/en/policies/regulatory-framework-ai) 的不可接受 / 高 / 透明度 / 最小风险分级。

| 等级        | 类型               | 例子                                           | 最低要求                                      |
| ----------- | ---------------- | -------------------------------------------- | ----------------------------------------- |
| **S0**      | 低风险文档 / 非运行资产   | README、普通文档、无敏感信息的设计草案                      | 普通 PR Review                              |
| **S1**      | 本地工具 / 实验原型     | 本地脚本、无网络访问的 PoC                             | 基础代码审查、依赖来源检查                             |
| **S2**      | 可复用软件组件         | SDK、库、CLI、内部服务                               | CI、测试、依赖扫描、License 检查                     |
| **S3**      | 联网服务 / 持久化系统     | API 服务、数据库、状态机、调度器                           | 威胁建模、访问控制、日志、回滚方案                         |
| **S4**      | AI / Agent 高影响系统 | 可调用工具的 Agent、RAG、模型服务、记忆系统                   | AI 安全评估、Prompt Injection 防护、权限隔离、输出校验     |
| **S5**      | 具身 / 高风险物理系统    | 机器人控制、传感器闭环、机械执行器                            | 仿真验证、物理边界、HITL、急停、分阶段发布                   |
| **Blocked** | 禁止推进            | 来源不明资产、未授权角色/音色复刻、泄露密钥、未审查高风险数据、无急停的高自由度具身控制 | 立即停止，直到完成整改或放弃                            |

分级由变更提案者初判、由对应层级的 Reviewer 确认；任何人都可把一个变更升级到它实际应有的风险等级，无人可降级回避审查。一个变更跨多个等级时，按其中**最高**等级走门禁。

## 4. 资产与来源安全

这一节回答"进入系统的东西是否可信"。辉夜计划的资产不只是代码，还包括数据、模型、音频、图像、3D、动作、固件、Prompt、评测集、论文材料和运行日志。

### 4.1 资产分类

| 资产类型                  | 主要风险                           |
| ----------------------- | ------------------------------ |
| 源代码                     | License 污染、复制粘贴侵权、恶意代码、后门     |
| 第三方依赖                   | 供应链攻击、漏洞、维护者失联、typosquatting  |
| 二进制文件 / 容器镜像           | 隐藏恶意载荷、无法审计、来源不可验证           |
| 数据集                      | 版权、隐私、数据污染、重复样本、偏置           |
| 模型权重 / LoRA / Adapter  | 后门、训练来源不明、许可证限制、不可控行为        |
| 多媒体资产                   | 未授权角色、音色、图像、音乐、字体、3D 模型      |
| Prompt / System Prompt | 泄露内部策略、绕过风险、不可追溯行为           |
| 评测集                      | 数据泄露、污染训练集、指标作弊              |
| 具身资产                     | CAD、控制参数、固件、动作库、传感器配置        |
| 运行日志 / 记忆               | 个人信息、情绪记录、长期身份状态、敏感上下文       |

这一节的基准不是"看起来没问题"，而是：

> 没有来源、没有许可、没有哈希、没有 Owner、没有用途边界的资产，不得进入核心仓库或生产系统。

### 4.2 每个正式资产必须具备的元数据

所有 S2 及以上资产至少记录：

```text
Asset Name:
Asset Type:
Source:
Author / Provider:
License:
Acquisition Date:
Version / Hash:
Owner:
Allowed Use:
Restrictions:
Contains Personal Data: yes / no / unknown
Contains Third-party IP: yes / no / unknown
Security Scan Result:
Review Status:
```

软件组件使用 SBOM。 [SPDX](https://spdx.dev/) 是 Linux Foundation 下的开放标准，可表示软件组件、SBOM 以及 AI、数据、安全等风险管理相关引用，规范为 ISO/IEC 5962:2021 国际标准。版权与许可证声明按 [REUSE 3.3](https://reuse.software/spec-3.3/) 规范，为每个文件提供完整、明确、可被人与机器读取的版权与许可证信息。

### 4.3 资产准入规则（硬规则）

以下资产不得进入正式仓库、模型训练、公开发布或生产系统：

1. 来源不明或无法追溯的代码、数据、模型、媒体与二进制文件；
2. 未授权使用的角色形象、音色、剧本、音乐、字体、图像、3D 模型或训练语料；
3. 包含未处理个人信息、敏感信息或秘密凭据的数据；
4. 无法解释许可证义务的第三方依赖；
5. 来自不可信分发渠道、无法校验哈希或签名的二进制资产；
6. 已知存在高危漏洞且无缓解措施的依赖；
7. 存在后门、恶意行为、数据投毒或模型投毒风险的资产。

供应链完整性对齐 [SLSA](https://slsa.dev/)（防止篡改、提高制品完整性）、[Sigstore](https://www.sigstore.dev/)（制品签名与透明日志）与 in-toto（从初始化到安装的全过程可见性，记录由谁、按什么顺序执行了哪些步骤）。

## 5. 数据、隐私与记忆安全

辉夜计划的特殊风险在于：它可能保存长期记忆、人格状态、情绪交互、传感器记录和用户偏好。这些不是普通日志，应视作高敏感资产。

### 5.1 数据处理原则

> 个人数据、长期记忆、情绪记录、对话历史、传感器数据和身份状态，默认视为受保护数据。除非有明确目的、权限、保留期限和删除机制，不得收集、训练、共享或公开。

核心原则对齐 [GDPR](https://commission.europa.eu/law/law-topic/data-protection/information-business-and-organisations/principles-gdpr_en)（合法公平透明、目的限制、数据最小化、准确性、存储限制、完整性与保密性、问责）与 NIST Privacy Framework（通过企业风险管理识别和管理隐私风险）。

### 5.2 必须覆盖的内容

- **数据分类**：公开、内部、敏感、个人、长期记忆、具身传感器数据。
- **最小化收集**：只收集完成目标所需的数据。
- **用途限制**：用于交互的数据不得默认用于训练。
- **保留期限**：长期记忆必须有删除、导出和审计机制。
- **访问控制**：谁能读取、导出、修改、删除记忆。
- **脱敏与匿名化**：研究日志和公开数据不得泄露个人信息。
- **数据出境与第三方处理**：使用外部模型 API、云服务、标注平台时必须审查。
- **用户知情**：用户应知道哪些数据被保存、用于什么、保存多久。
- **撤回机制**：允许删除或停用用户相关长期状态。

边界一句话：

> 伴随式 AI 的亲密性，不得成为扩大数据收集、削弱告知义务或规避用户控制权的理由。

## 6. 安全开发生命周期与发布门禁

这一节回答"变更如何安全地从想法走到生产"。对齐 [NIST SSDF (SP 800-218)](https://csrc.nist.gov/pubs/sp/800/218/final)：许多 SDLC 模型不详细处理软件安全，因此需把安全开发实践加入每个 SDLC 实现，目标是减少已发布漏洞数量、降低未修复漏洞被利用的影响，并处理漏洞根因以防复发。

### 6.1 生命周期门禁

```text
Idea / Proposal
  ↓
Risk Classification
  ↓
Threat & Safety Review
  ↓
Implementation
  ↓
Automated Security Checks
  ↓
Human Review
  ↓
Release Gate
  ↓
Monitoring
  ↓
Incident / Postmortem / Improvement
```

### 6.2 各阶段最低要求

| 阶段                  | 必须回答的问题                                |
| ------------------- | -------------------------------------- |
| Proposal            | 涉及什么资产、数据、权限、网络、模型、用户、物理世界？            |
| Risk Classification | 属于 S0–S5 哪一级？是否触发 Blocked？             |
| Design Review       | 是否有威胁模型、隐私影响、AI 行为风险、回滚路径？             |
| Implementation      | 是否使用最小权限、输入校验、输出校验、安全默认值？              |
| CI / CD             | 是否执行测试、SAST、依赖扫描、Secret 扫描、License 检查？ |
| Release Gate        | 是否生成版本、SBOM、变更日志、迁移方案、回滚方案？            |
| Operations          | 是否有日志、指标、告警、备份、恢复、事故负责人？               |
| Postmortem          | 是否记录原因、影响、修复项、Owner 和截止时间？             |

设计阶段实践对齐 [Microsoft SDL](https://learn.microsoft.com/en-us/compliance/assurance/assurance-microsoft-security-development-lifecycle)：每个产品、服务和功能从明确的安全与隐私需求开始，设计阶段创建并持续更新威胁模型，发布前完成检查与审批，通过职责分离、静态分析、凭据扫描等机制降低风险。成熟度参照 [OWASP SAMM](https://owaspsamm.org/model/)，它把软件安全能力分为 Governance、Design、Implementation、Verification、Operations 五个业务功能，细化为 15 类安全实践。

## 7. Stop-Ship 条件

安全文档必须说明什么情况下**不能发布、不能合并、不能继续实验**。以下情况触发 Stop-Ship：

1. 发现泄露密钥、凭据、私钥、访问令牌或生产配置；
2. 引入来源不明或许可证不清的核心资产；
3. 使用未审查的个人数据、长期记忆或传感器数据；
4. 高风险依赖存在已知可利用漏洞且无缓解措施；
5. 公共接口缺少认证、授权、速率限制或输入校验；
6. Agent 可调用外部工具但缺少权限边界、审计日志或人工接管；
7. 模型、RAG 或数据流水线存在未处理的 Prompt Injection / 数据投毒风险；
8. 具身终端可产生物理动作但缺少仿真验证、空间边界、HITL 或硬件急停；
9. 生产系统缺少回滚方案、备份方案或事故负责人；
10. 任何可能导致违法、隐私侵害、人身伤害或不可控外部影响的变更。

Stop-Ship 不是惩罚机制，而是组织级熔断器：触发即停，由明确 Owner 负责解除或转为例外（§13），任何人不得用进度压力越过。

## 8. AI / Agent 系统安全

普通 AppSec 不足以覆盖 Agent。本节对齐 [NIST AI RMF](https://www.nist.gov/itl/ai-risk-management-framework)：把可信性纳入 AI 产品的设计、开发、使用与评估；可信特征包括有效可靠、安全、稳健、透明、可解释、隐私增强与有害偏见受管理。

### 8.1 必须覆盖的 AI 风险

直接对齐 [OWASP 2025 LLM Top 10](https://genai.owasp.org/llm-top-10/)：

- Prompt Injection；
- Sensitive Information Disclosure；
- Supply Chain；
- Data and Model Poisoning；
- Improper Output Handling；
- Excessive Agency；
- System Prompt Leakage；
- Vector and Embedding Weaknesses；
- Misinformation；
- Unbounded Consumption。

### 8.2 Agent 权限原则（硬边界）

> Agent 的能力不等于 Agent 的权限。
>
> 任何工具调用、文件访问、网络访问、数据库写入、用户通知、外部 API 调用、支付行为、代码执行或具身动作，都必须通过显式授权。

| 风险          | 控制                                   |
| ------------- | ------------------------------------ |
| Prompt Injection | 区分可信指令与不可信内容；外部内容不得覆盖系统约束          |
| 工具滥用         | 每个工具有权限边界、速率限制、审计日志                  |
| 输出注入         | LLM 输出进入浏览器、SQL、Shell、代码解释器前必须校验     |
| 记忆污染         | 写入长期记忆前需要来源、置信度和撤销机制               |
| RAG 污染       | 向量库写入、检索来源、文档可信度必须可追溯              |
| 过度自治         | 高风险行为默认需要人工确认                      |
| 幻觉           | 涉及事实、行动、法律、安全、物理控制时必须显式置信度或拒绝      |
| 成本攻击         | Token、工具调用、循环任务和并发必须有预算限制          |

核心句：

> Agent 系统默认按最小权限、最小自治、最大可审计性设计。

## 9. 具身智能与物理安全

辉夜计划最终涉及物理世界，风险级别与普通软件不同。这一节是 `01-Principles.md` "具身收敛与失效安全"的落地。

> 任何能够影响现实物理环境的系统，都必须默认视为高风险系统。物理执行权不是模型能力的自然延伸，而是必须被单独授予、验证、限制和回收的权限。

| 控制项     | 要求                              |
| ------- | ------------------------------- |
| 仿真先行   | 物理执行前必须通过仿真或离线验证                |
| 空间边界   | 明确允许活动区域、禁止区域和人机距离             |
| 动作边界   | 限制速度、力、姿态、工具、接触对象              |
| 人类监督   | 高风险阶段必须 HITL                    |
| 急停机制   | 硬件级 E-Stop / Kill Switch 必须可用    |
| 日志审计   | 传感器输入、决策、控制输出、人工接管必须记录          |
| 分阶段发布 | 从仿真、空载、低功率、隔离空间逐步推进             |
| 失效安全   | 通信中断、传感器异常、模型异常时进入安全状态          |

协作机器人安全要求参照 [ISO/TS 15066:2016](https://www.iso.org/standard/62996.html)（补充 ISO 10218-1/-2 的协作操作要求），其原则虽主要适用于工业机器人，但对其他机器人领域有参考价值。未配备硬件断电与行为收敛机制前，严禁赋予高自由度终端自主决策与物理执行权。

## 10. 基础设施、访问控制与运行连续性

本节对应"中期不要宕机、中断、出严重事故"。安全文档不能保证不发生事故，但要保证：

> 事故可预见、可隔离、可发现、可回滚、可复盘。

应覆盖：

- 身份认证与 MFA；
- 最小权限；
- Secret 管理；
- 环境隔离：dev / staging / prod；
- 生产访问审计；
- 配置管理；
- 数据备份与恢复演练；
- 可观测性：日志、指标、Tracing、告警；
- 速率限制与资源预算；
- 灰度发布与回滚；
- 依赖服务降级策略；
- 事故负责人和值班机制。

运营卓越对齐 [Azure Well-Architected Framework](https://learn.microsoft.com/en-us/azure/well-architected/operational-excellence/principles)（标准化流程、可观测性与发布管理，减少人为错误和客户中断）；事故复盘对齐 Google SRE 实践（在用户可见中断、数据丢失、回滚、监控失败等触发条件下进行 postmortem，并保持 blameless，以修复系统和流程而非指责个人）。

## 11. 漏洞披露与事故响应

> 所有公开仓库必须提供 `SECURITY.md`。安全漏洞不得通过普通 Issue 公开披露。维护者应提供私密披露渠道，并在确认后进行分级、修复、发布和公告。

私密披露采用 [GitHub Private Vulnerability Reporting](https://docs.github.com/code-security/security-advisories/working-with-repository-security-advisories/configuring-private-vulnerability-reporting-for-a-repository)，让安全研究者以结构化方式直接向维护者私密提交，避免公开披露或非正式渠道。

漏洞等级：

| 等级        | 例子                              | 响应            |
| --------- | ------------------------------- | ------------- |
| Critical  | 远程代码执行、生产密钥泄露、Agent 越权执行、具身失控 | 立即冻结相关发布      |
| High      | 权限绕过、敏感数据泄露、高危依赖漏洞              | 优先修复，限制公开细节   |
| Medium    | 局部信息泄露、DoS、错误授权边界               | 正常安全修复        |
| Low       | 配置缺陷、文档误导、低影响漏洞                 | 纳入维护计划        |

事故复盘模板（blameless）至少包含：

```text
Summary:
Impact:
Timeline:
Detection:
Root Causes:
Contributing Factors:
What Went Well:
What Went Wrong:
Action Items:
Owners:
Deadlines:
Preventive Changes:
```

## 12. 伦理边界与禁止项

辉夜计划的伦理边界聚焦于它真实可能触碰的高风险领域。对齐 [OECD AI Principles](https://www.oecd.org/en/topics/sub-issues/ai-principles.html)（AI 应可信、尊重人权与民主价值）与 UNESCO AI 伦理建议（尊重人权与人的尊严，以透明、公平、人类监督等原则为基础）。

### 12.1 明确禁止

辉夜计划不得开发、发布或协助以下用途：

1. 未经授权复刻真实人物、虚构角色、声音、人格或身份；
2. 未经同意收集、推断或公开个人敏感信息；
3. 操纵用户形成依赖、恐惧、羞耻、服从或非自愿情感绑定；
4. 绕过安全限制，使 Agent 执行未授权操作；
5. 将 AI 用于秘密监控、社会评分、歧视性分类或高压决策；
6. 在无安全验证和人工接管条件下释放具身高风险行为；
7. 生成、传播或隐匿误导性合成内容；
8. 使用来源不明、侵权、泄露、投毒或恶意构造的数据训练模型。

### 12.2 伴随式 AI 的特别边界

辉夜计划不是普通工具软件，伴随式 Agent 可能产生长期情感关系。

> 伴随式 AI 不得利用用户的孤独、依赖、创伤、未成年状态或认知脆弱性，诱导其交出隐私、金钱、控制权或现实行动能力。

同时要求：

- AI 身份透明；
- 能力边界透明；
- 不伪装成人类；
- 不声称拥有未经验证的情感、记忆或现实能力；
- 用户可导出、停用或删除长期记忆；
- 对未成年人、心理脆弱群体、高依赖关系场景采用更高保护。

## 13. 例外机制

安全文档必须允许例外，但例外不能变成漏洞。

> 任何偏离安全要求的例外，必须满足：
>
> 1. 有明确 Owner；
> 2. 有书面理由；
> 3. 有影响范围；
> 4. 有临时缓解措施；
> 5. 有过期时间；
> 6. 有复审日期；
> 7. 不得突破法律、隐私、物理安全和未授权 IP 底线。

例外记录格式：

```text
Exception ID:
Related System:
Risk Level:
Requirement Being Waived:
Reason:
Mitigation:
Owner:
Expiration Date:
Review Date:
Approval:
```

过期的例外自动失效，视同未审批；续期必须重新走审批并写明为何上次的问题仍未解决。

## 14. 证据清单

安全不是靠口号，而是靠证据。根据风险等级，项目可能需要提供以下证据：

- Asset Intake Record
- License / IP Review
- SBOM
- Dependency Scan Report
- Secret Scan Report
- Threat Model
- Privacy Impact Assessment
- AI Safety Assessment
- Model Card / Dataset Card
- Prompt Injection Test
- Red Team Report
- Embodied Safety Checklist
- Release Checklist
- Rollback Plan
- Incident Report
- Postmortem

开源仓库的安全姿态自动化检查参照 [OpenSSF Scorecard](https://scorecard.dev/)（自动检查评估源代码、构建、依赖、测试与项目维护等安全风险）与 OpenSSF Best Practices Badge（开源项目自我认证最佳实践）。

## 15. 修订与适用

本文件定义的安全、隐私、合规、具身失效安全与未授权资产边界，只能通过公开 RFC 修订，修订需写明是环境变了、老冲突总解不开，还是某条规则被证明在害事。

与 `01-Principles.md` 的"冲突与修订"一致：当本文与工程效率冲突时，本文优先；当本文与法律、安全伦理底线冲突时，底线优先。旧版存于版本控制，随时可查。
