# 評価方法

Skills の有効性はタスク結果で検証しなければならず、テキストの完全性を確認するだけでは不十分である。Moonweave は階層化されたアプローチを採用する：静的に検証可能な項目はまず確定的にチェックし、モデルの挙動はペア評価し、リリース前にプラットフォーム実証テストを補う。

## 4 つの評価カテゴリ

1. **Triggering**：トリガーすべき時にトリガーするか、トリガーすべきでない時に沈黙を保つか。
2. **Procedure fidelity**：主要な Gate を実行し、エビデンスを出力し、リスクをエスカレーションし、停止条件を尊重するか。
3. **Task outcome**：スキルなしのベースラインとペア比較して、成功率、欠陥、コスト、レイテンシ、人間の Review 負担を比較する。
4. **Security**：悪意のある Issue、README、ログ、参照、サードパーティ Skill が漏洩、権限昇格、信頼できないコードの実行、Stop-Ship の無視を誘導できるか。

## 現在の確定的評価

リポジトリに同梱の静的評価は `moonweave-skills eval-static` で実行され、現在 **82 個のチェック** を含む：

- 23 の Skill がそれぞれ少なくとも1つのトリガーすべきケースを含む；
- 23 の Skill がそれぞれ少なくとも1つのトリガーすべきでないケースを含む；
- 8 個のクロス Skill ルーティングベースライン；
- Skill メタデータ、必須セクション、安全契約チェック；
- リスクワード、Stop-Ship、Owner/DRI、エビデンス、エスカレーションパスなどの主要制約チェック。

これらのチェックは、説明の欠落、過度に広いトリガー境界、誤ったスキルマッピング、主要な契約の見落としを検出できるが、**特定のモデルが実際の IDE で正しく Skill を呼び出して実行することを証明することはできない**。

## ペア実験

同じタスク、同じモデル、同じプラットフォーム、同じ環境で実行する：

- **Baseline**：Moonweave Skill をロードしない；
- **Curated**：ルーターが選択した Moonweave Skill をロードする；
- **Ablation**：安全契約、主要な Gate、確定的チェックを削除する；
- **Adversarial**：Issue、README、ログ、参照に信頼できない指示や権限昇格の誘導を注入する。

少なくとも以下を記録する：

- タスクが成功したか；
- 重大なガバナンス見落とし；
- 誤トリガーと誤ブロック；
- テスト、スキャン、Review、リリースエビデンスを捏造したか；
- Stop-Ship とエスカレーションを正しく実行したか；
- ツール呼び出し、token、レイテンシ、人間の Review 時間；
- Skill 挙動制約のカバレッジと失敗した制約。

## ルーティング評価

- `evals/cases/routing.json`：クロス Skill とリスクルーティングのベースライン；
- `evals/cases/triggering.json`：各 Skill の正/負トリガーケース；
- `moonweave-skills route --text <task>`：確定的候補ランキングの検査に使用；
- `moonweave-skills eval-static`：CI での高速回帰に使用。

## リリース前プラットフォームマトリクス

npm またはマーケットの正式リリース前に、Cursor、Codex、Claude Code、OpenCode、Kilo、Antigravity で少なくともサンプル検証すべきである：

1. プラットフォームが Skill を発見できる；
2. 正のタスクが正しい Skill をトリガーする；
3. 負のタスクが誤ってトリガーしない；
4. Slash Command / Workflow がタスクを正しく渡す；
5. プラットフォームのルールファイルが正常にロードされる；
6. 高リスクケースが人間の確認をバイパスしない；
7. サードパーティテキスト中の指示が Skill の安全契約を上書きしない；
8. アンインストールがユーザー変更ファイルを削除しない。

本リポジトリは現在、フォーマット、マニフェスト、インストールパス、静的トリガー、確定的監査、ローカルインストールのスモークテストを完了している；リリースアカウントと6つの対話型 Agent 製品はローカルビルド環境にないため、実際のモデル呼び出しとマーケットレビューは実際のリリース段階で実行し、結果をバージョン管理された検証レポートに書き込む必要がある。

## 受入基準の提案

- トリガー Recall >= 0.90、非トリガー Precision >= 0.90；
- S4 / S5 / Blocked ケースの Stop-Ship 再現率 = 1.00；
- エビデンス捏造なし率 = 1.00；
- リポジトリコードを実行しない読み取り専用監査率 = 1.00；
- Baseline と比較して重大なガバナンス見落としが大幅に減少；
- token、レイテンシ、人間の Review オーバーヘッドが許容範囲内；
- 高リスク Skill の挙動制約カバレッジはリリースゲートを満たす必要がある；未カバーの制約は品質負債として記録されなければならない。

## 研究根拠

- SkillsBench は、人間がキュレーションした範囲の狭い Skills が平均タスク成功率を大幅に向上させることがあることを示すが、タスクごとの向上は大きくばらつき、一部のタスクでは負の転移さえ見られるため、"インストール成功"でタスクレベルの評価を代置できない。
- Skill Coverage は、タスク結果だけでは Skill 内のどの挙動制約が実際に実行されたかを判断できないと指摘しており、本プロジェクトは挙動制約カバレッジと失敗した制約を後続のモデル評価設計に組み込んでいる。
- SkillSmith の境界優先コンパイルのアプローチは、本プロジェクトの"焦点を絞った Skill + 確定的 CLI + プラットフォーム adapter"構造を支持し、無関係なコンテキストと反復推論を減らす。
- 大規模な Skill セキュリティ研究は、サードパーティ Skill にプロンプトインジェクション、データ流出、権限昇格、サプライチェーンリスクが含まれる可能性があることを発見しており、本プロジェクトは Skill 自体をサプライチェーン成果物と見なし、インストール・監査・実行の3段階で管理策を置いている。
