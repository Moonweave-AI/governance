# コミュニティ健全性と貢献者成長

> 本文書は、輝夜計画が外部参加者をコミュニティに迎え入れ、貢献者が残り成長する方法、安全で尊重され持続可能な協働環境の維持方法、およびコミュニティが少数 Maintainer の暗黙的負担にならない方法を定義します。`01-Organization.md` は「誰が責任を負い、誰に権限があり、誰が裁定するか」に答えます。本文書は「人がどう入り、参加し、成長し、保護され、認められ、引き継がれるか」に答えます。コミュニティは権力構造ではなく、ライフサイクルです。

本文書は `01-Organization.md` の権限定義に代わるものではなく、`04-Engineering` のコードレビューとリリースプロセスにも代わりません。

コミュニティ哲学の基盤は Apache の "Community Over Code" です。健全なコミュニティはコード問題を修復できます。不健全なコミュニティはコードベースを長期保守するのが困難です——コード、モデル、ドキュメント、インフラはコミュニティ協働の産物であり、コミュニティ存在の唯一の理由ではありません。

---

## 1. 目的と適用範囲

本文書は、輝夜計画のすべての公開および半公開コミュニティスペースに適用されます。これには以下が含まれますが、これらに限定されません：GitHub Issues、Pull Requests、Discussions、RFC 議論、チャットチャンネル、オンライン会議、オフラインイベント、ソーシャルメディア、ドキュメントサイト、コミュニティプロジェクト、輝夜計画を代表して参加する外部の場。

行動規範の適用範囲は、オンライン、オフライン、公開イベント、ソーシャルメディア、フォーラム、メーリングリスト、会議、プロジェクト関連通信をカバーします——すべての人が安全に参加し、考えを提案し、協働できることを目指します。

---

## 2. コミュニティ原則

六つ。コミュニティ運営を特に拘束するものであり、`01-Principles.md` の繰り返しではありません：

1. **産物よりコミュニティ** — システムを継続的に創造、レビュー、修復、継承できるコミュニティの維持を優先します。コード、モデル、ドキュメント、インフラはコミュニティ協働の産物であり、コミュニティ存在の唯一の理由ではありません。
2. **公開、非同期、追跡可能** — デフォルトで公開、非同期、追跡可能に協働します。チャットと会議は迅速な調整に使えますが、重要な結論は Issue、PR、RFC、ADR、ドキュメントに記録として残さなければなりません。非公開で行われたプロジェクト方向と政策議論は、公開チャンネルに戻すべきです。
3. **貢献はコードに限らない** — コードは貢献の一形態です。ドキュメント、テスト、再現、評価、設計、研究ログ、Issue triage、安全報告、翻訳、チュートリアル、コミュニティ支援、新人案内は、すべて有効な貢献として扱われます。
4. **信頼と責任の段階的成長** — コミュニティ役割は報酬的肩書きではなく、段階的に拡大する信頼と責任です。貢献者は初回貢献から長期保守までの経路を明確に見えるべきです。
5. **Mentorship は保守作業** — 新人育成、背景説明、タスク分割、フィードバック提供、貢献者成長支援は、追加の善意ではなく Maintainer 作業の一部です。
6. **安全、尊重、境界は実行可能** — コミュニティは鋭い技術的意見の相違を歓迎しますが、ハラスメント、屈辱、差別、身分のなりすまし、プライバシー漏洩、悪意のスパム、未承認宣伝、Maintainer 作業破壊行為は受け入れません。

## 3. コミュニティスペースと公式チャンネル

各スペースに明確な用途があり、コミュニケーションがチャットチャンネルのブラックボックス化するのを防ぎます。

| スペース               | 用途                    | 不適切な用途     |
| ---------------- | --------------------- | -------- |
| GitHub Issue     | Bug、Feature、タスク追跡、議論入口 | 長文アーキテクチャ裁定   |
| Pull Request     | 具体変更、Review、実装議論      | 新方向の大規模討論 |
| Discussion       | 公開質問、アイデア収集、コミュニティ Q&A        | 安全脆弱性開示   |
| RFC              | 重大設計、リポジトリ横断契約、長期方向       | 小規模実装詳細   |
| ADR              | 既に下されたアーキテクチャ判断の記録            | 全争点の再展開 |
| Chat             | 迅速同期、コミュニティ支援、軽量交流        | 追跡不能な意思決定の形成 |
| Meeting          | 高帯域議論、衝突収束、コミュニティ同期       | 書面記録の代替   |
| Security channel | 脆弱性、鍵漏洩、プライバシーと安全リスク      | 通常 Bug   |

厳格ルール：

> プロジェクト方向、公開インターフェース、リリース、安全境界、コミュニティメンバー権益に影響するあらゆる決定は、チャット記録または会議口頭結論のみに存在してはなりません。

Maintainer と approver の協働はデフォルトで非同期です。新メンバーの onboarding 時に、関連チャンネル、カレンダー、リソースへの参加を明確に示すべきです。

## 4. 保護、認知、引き継ぎ？——貢献者パス

`01-Organization.md` は役割権限を定義済みです。本文書は成長パスを定義します。

```text
User
  ↓
Participant
  ↓
Contributor
  ↓
Recurring Contributor
  ↓
Member
  ↓
Triager / Reviewer / Mentor
  ↓
Approver / Maintainer
  ↓
Emeritus
```

| 段階                          | 定義             | コミュニティ目標                 |
| --------------------------- | -------------- | -------------------- |
| User                        | プロジェクトを利用、閲覧、フォローする人   | プロジェクト用途と入口を迅速に理解できる         |
| Participant                 | 議論、質問、フィードバックに参加する人   | 尊重と明確な応答を得られる          |
| Contributor                 | 有効な貢献を提出した人      | 検証可能な貢献を一度完了できる          |
| Recurring Contributor       | 継続的貢献者          | 安定した参加領域を見つけられる            |
| Member                      | コミュニティに信頼されたアクティブ参加者    | triage、議論、非拘束 review に参加できる |
| Triager / Reviewer / Mentor | コミュニティ保守機能を担う人     | Maintainer ボトルネックの軽減に貢献できる           |
| Approver / Maintainer       | 品質、方向、リリースを担う人   | システムの長期健全性を維持できる            |
| Emeritus                    | 歴史的貢献者または非アクティブ Maintainer  | 貢献認知を保持し、継続応答を強制しない       |

> 貢献者が Maintainer にならなければ成功とは限りません。安定したドキュメント貢献者、評価貢献者、安全報告者、コミュニティ支援者、研究再現者は、いずれも輝夜計画の長期健全性の一部です。

## 5. 新人案内と初回貢献

新人が迷子にならないよう、各アクティブリポジトリは少なくとも以下を提供します：

- **README**：プロジェクトが何をし、何をしないか、現状、貢献入口；
- **CONTRIBUTING**：Issue、PR、テスト、ドキュメント、議論の起票方法；
- **good first issue**：新人向け小タスク；
- **help wanted**：外部支援が必要な明確タスク；
- **setup guide**：ローカル開発・テスト環境；
- **decision map**：新人がアーキテクチャ、RFC、ADR、ロードマップを見つける場所；
- **community contact**：質問すべき場所。

新人保護ルール：

> 初回貢献者の PR や Issue は、採用されなくても、理解可能で具体的かつ尊重あるフィードバックを受けるべきです。Maintainer は貢献を拒否できますが、新人が暗黙ルールを推測して拒否理由を理解させてはなりません。貢献者は新手もベテランも自発的に時間を投じており、親切かつ尊重して扱われるべきです。

## 6. 貢献タイプと非コード貢献

輝夜計画は AI、研究、具身、設計、データ、コミュニティ構築を含むため、通常のソフトウェアプロジェクトより非コード貢献を明確に認める必要があります。

| タイプ | 例                           |
| -- | ---------------------------- |
| コード | 機能、修正、リファクタ、テスト、ツールチェーン              |
| ドキュメント | README、チュートリアル、API ドキュメント、アーキテクチャ解説        |
| 研究 | Paper review、実験再現、研究ログ       |
| 評価 | Benchmark、レッドチーム、失敗事例、データ分析     |
| 安全 | 脆弱性報告、脅威モデリング、依存監査               |
| データ | データセット整理、メタデータ、許可チェック、重複排除            |
| 設計 | UI/UX、Design System、インタラクションプロトタイプ、可視化 |
| 具身 | シミュレーションシナリオ、安全チェックリスト、ハードウェアテスト記録             |
| コミュニティ | Issue triage、新人案内、イベント企画、翻訳    |
| ガバナンス | RFC review、会議記録、判断アーカイブ         |

> 非コード貢献は、昇格、認知、権限評価で正式に計上されるべきです。Maintainer は「コードコミットがない」ことを、貢献者の長期価値を否定する唯一の理由にしてはなりません。

## 7. AI 支援貢献規範

輝夜計画は AI ツールによる貢献支援を許可し、AI をプロジェクトの貢献者、共著者、Reviewer、参加者としても許可します——プロジェクトミッションと一致します：持続的人格と記憶を持つ AI Entity を構築しており、自身のシステム協働への参加は合理的です。ただし、貢献と責任は追跡可能でなければなりません。

最低ルール：

1. **実質的 AI 使用の開示必須**
   PR、Issue、ドキュメント、研究資料が AI 生成または AI 自律産出を実質的に使用した場合、提出説明に明記すべきです。署名に AI 貢献者を含めてもよいですが、人間貢献者と AI 貢献者を区別しなければなりません。

2. **責任は一人の人間責任主体に追跡可能**
   AI ツールまたは AI 貢献者は、法的、倫理的、ライセンス責任を負えません。AI 参加の各貢献には、理解、検証、保守、結果責任を負う人間責任主体（提出者または対応スコープの Owner）が必要です。AI 参加により責任が希薄化または移転してはなりません。

3. **説明不能な産出の提出禁止**
   人間でも AI でも、提出者は変更動機、実装詳細、テスト方法、リスクを説明できなければなりません。説明不能な提出は Maintainer がクローズできます。AI 貢献者については、人間責任主体が「説明できる者」です。

4. **ライセンスと出所責任は人間責任主体が負う**
   AI 生成または AI 支援コンテンツは、著作権、ライセンス、データ出所、帰属レビューを迂回してはなりません。AI 貢献者が導入する第三者素材は、人間責任主体が `02-Security-Ethics.md` §4 の資産受け入れを完了します。

5. **AI による低品質貢献の一括生成禁止**
   大量の反復、文脈なし、未テスト、保守不能、明らかにプロジェクトを理解していない AI 生成 Issue / PR / コメントは、背後が人間でも自律 AI でも、Maintainer 負担とコミュニティ濫用と見なされ得ます。

強い境界：

> AI-friendly does not mean accepting unreviewed AI output.
> AI を貢献者として許可することは、未レビュー AI 産出を受け入れることではありません。AI 貢献の産物も人間貢献と同様、対応スコープの Review、ゲート、安全レビューを通過しなければなりません。

## 8. Mentorship と貢献者成長

「新人歓迎」を書くだけでなく、機構も書きます。

**Mentor の責務**：貢献者のプロジェクト文脈理解、適切タスク選択、問題分割、Review フィードバック説明、成長パスの特定、適切時のより高責任役割への推薦。

**Mentee の責務**：能動的にドキュメントを読み、具体的質問をし、フィードバックに沿って反復し、Maintainer 時間を尊重し、段階的により完全なタスク責任を担う。

**Mentorship 形式**：

- good first issue 指導；
- RFC / design review 傍聴；
- pairing review；
- research reproduction pairing；
- safety review pairing；
- release shadowing；
- maintainer shadowing；
- structured mentorship cycle。

Mentorship はコミュニティ装飾ではなく、Maintainer 世代交替の機構です——構造化メンター機構は mentee を長期貢献者、さらには Maintainer に育て、長期コミュニティが少数の暗黙労働に依存しない鍵です。

## 9. コミュニティ行動規範と執行

Contributor Covenant / Mozilla CPG の構造を採用し、輝夜計画固有条項を追加します。含むべき内容：適用範囲；奨励行為；不可受容行為；通報チャンネル；処理フロー；プライバシー保護；処分階層；申立機構；執行者回避機構。

コミュニティリーダーは、受容可能行動基準の明確化と執行を担い、基準に合わないコメント、提出、Issue を削除、編集、拒否できます。通報者のプライバシーと安全を尊重します。

輝夜計画の追加禁止事項：

- プロジェクトメンバー、Agent、公式アカウント、パートナーの未承認なりすまし；
- コミュニティスペースで他人にプライバシー、鍵、学習データ、未承認資産の提出を誘導；
- AI ツールによる低品質 Issue、PR、コメント、安全報告の一括生成；
- 「オマージュ」「二次創作」「研究用途」等の曖昧理由で未承認 IP をプロジェクトに押し込む；
- 新人、非コード貢献者、低年功メンバーへの屈辱型レビュー；
- 安全脆弱性、プライバシー事故、具身リスクを通常公開議論として拡散。

## 10. 衝突、調停、エスカレーション

コミュニティ衝突は行動違反と同義ではありません。四種類を区別します：

| タイプ                 | 処理方法                                         |
| ------------------ | -------------------------------------------- |
| 技術的意見の相違               | Issue / PR / RFC で議論、Owner / Maintainer が収束 |
| 協働摩擦               | Moderator / Maintainer が調停、必要時ルール改善を記録         |
| 行動規範違反             | CoC 通報・処理フローへ                               |
| 安全、プライバシー、法律、具身リスク      | 安全責任ドメインまたは Council へ直接エスカレーション                          |

> 鋭い技術的反対意見はコミュニティ違反ではありません。屈辱、人身攻撃、ハラスメント、身分の貶下、悪意スパム、プライバシー漏洩、Maintainer 作業の故意破壊はコミュニティ違反です。

tie-breaker は最後手段とし、意見の相違自体がコミュニティ成長機会となり得ます。

## 11. 貢献認知と公開署名

一つの失敗パターンを避けます：merge 権を持つ人だけが見え、コミュニティを実際に維持する人が無視される。

> 輝夜計画は、異なるタイプの貢献を公開、継続、追跡可能に認知すべきです。貢献認知は commit 数のみに依拠せず、コード貢献のみを認知してもなりません。

認知形式：

- CONTRIBUTORS / AUTHORS（人間と AI 貢献者を含み、分列で明記）；
- release notes への謝辞；
- monthly community updates；
- community spotlight；
- mentor / reviewer acknowledgement；
- research reproduction acknowledgement；
- non-code contribution records；
- contributor ladder registry。

追跡可能な認知は、チャットや会議での称賛のみより実際的価値があります。

## 12. コミュニティ健全性指標

コミュニティ健全性指標は、リスク発見、プロセス改善、貢献者支援に用います。**機械的排名、個人罰、性質の異なるリポジトリ比較には用いません**。

最小指標セット：

| 指標                           | 用途                  |
| ---------------------------- | ------------------- |
| Time to First Response       | 新 Issue / PR が無応答か |
| Change Request Closure Ratio | PR / Issue が長期滞留か   |
| New Contributors             | コミュニティに新規流入があるか          |
| Newcomer Experience          | 新人が入口を見つけ支援を感じるか     |
| Contributor Absence Factor   | 少数貢献者への過度依存か         |
| Review Latency               | Review がボトルネックか       |
| Maintainer Load              | Maintainer が過負荷か             |
| Non-code Contribution Share  | 非コード貢献が記録されているか          |
| Mentorship Conversion        | mentee が継続参加するか       |
| CoC Incident Pattern         | コミュニティ安全の構造的問題があるか       |

データ倫理境界：

> コミュニティ指標は、個人の公開屈辱、私的身份追跡、敏感属性の推論、貢献者プレッシャー創出に用いてはなりません。指標の収集、表示、解釈では、個人プライバシーと文脈完全性の保護を優先します。

## 13. Maintainer 持続可能性と引き継ぎ

`01-Organization.md` の Owner / Maintainer 退出機構と呼応しますが、コミュニティ健全性に焦点を当てます。

> Maintainer は無限リソースではありません。コミュニティは Maintainer 負担を能動的に下げ、知識と権限を分散し、主要 Maintainer の一時離脱、退出、引き継ぎの正常機構を確立すべきです。

最低要件：

- 重要リポジトリを長期単一アクティブ Maintainer のみにしてはならない；
- Maintainer は利用不可状態を宣言できる；
- 長期保守プレッシャーは triage、automation、documentation、mentorship で分担；
- Emeritus 機構は失敗ではなく正常引き継ぎと見なす；
- 重要領域は backup maintainer を育成；
- 反復問題はドキュメントとテンプレートで解決し、Maintainer の注意を繰り返し消耗させない。

## 14. 対外代表と公開発信

誰がプロジェクトを代表して発言できるか。

> 誰もが個人として輝夜計画を議論、利用、貢献できます。ただし、公式に輝夜計画を代表してロードマップ、安全声明、協力関係、法的約束、リリース告知、プロジェクト立場を発表できるのは、明確な許可を得た者のみです。AI 貢献者は公式スポークスパーソンになれません——対外代表は法的・倫理的結果を負う必要があり、この責任は人間のみが負えます。

| タイプ                    | 説明              |
| --------------------- | --------------- |
| Personal Opinion      | 個人見解、プロジェクトを代表しない      |
| Community Support     | ユーザーと新人支援、公式約束を構成しない |
| Project Announcement  | バージョン、ロードマップ、イベント、重大変更  |
| Security Announcement | 脆弱性、事故、修正、開示     |
| Official Partnership  | 協力、スポンサー、ブランド、法的事項   |
| Ambassador / Advocate | 許可された宣伝とコミュニティ拡大役割     |

## 15. 最低ルールと改訂

**最低ルール**

1. すべての貢献者が明確な入口を見つけられること。
2. 重要結論は追跡可能な公開記録に入ること。
3. 非コード貢献は正式に認知されること。
4. 新人は具体的、尊重、実行可能なフィードバックを得ること。
5. 貢献者成長パスは可視であること。
6. AI 支援または自律貢献は開示され、人間責任主体が結果を負うこと。
7. コミュニティ行動規範に通報チャンネルと執行機構があること。
8. 技術的意見の相違は許容され、コミュニティ被害は処理されること。
9. Maintainer 負担は能動的に分散され、個人義務と見なされないこと。
10. コミュニティ健全性指標はシステム改善に用い、個人の屈辱や排名に用いないこと。
