# 品質保障規範

> 本文書は、輝夜計画におけるソフトウェア、AI Agent、フロントエンド／バックエンドシステム、インフラストラクチャ、データパイプライン、モデルサービス、研究実験、具身制御システム、公開資産の品質保障体系を定義します。品質保障はリリース前のテスト段階ではなく、設計、実装、レビュー、ビルド、リリース、運用、フィードバック、退役を貫く継続的証拠システムです。エンジニアリング成果物は、品質証拠が対応するリスク等級とライフサイクルコミットメントを支えるに足る場合にのみ、マージ、リリース、本番投入、公開、長期保守に入ることができます。

本文書は以下に代わるものではありません：

- `01-Workflow.md`：エンジニアリング作業の進め方を定義；
- `01-Foundation/02-Security-Ethics.md`：セキュリティ、プライバシー、倫理の硬いボトムラインを定義；
- `standards/*`：各技術スタックの具体的実装基準を定義。

上記文書との関係：Workflow はプロセスノードを定義し、Security-Ethics は突破不可の境界を定義し、Quality-Assurance は**各ノードで、システムが正しい、信頼できる、安全、保守可能、可観測、リリース可能、運用可能、退役可能であることを証明するに足る証拠**を定義します。

---

## 1. 目的と適用範囲

本文書は 1 つの核心問題に答えます：輝夜計画のエンジニアリング産出は、何をもって「デリバリー可能」とみなされるのか？

答えは感覚でも、口頭確認でも、「ローカルで動いた」でもありません——検査可能、追跡可能、自動検証可能な証拠です。この証拠体系は設計レビューから運用モニタリングまでの全ライフサイクルをカバーし、リスク等級に応じて異なる証拠強度を割り当てます。

輝夜計画のすべての正式リポジトリにおけるコード、サービス、データ、モデル、Agent、プロトコル、フロントエンド／バックエンドシステム、インフラストラクチャ、具身制御システム、公開リリース資産に適用されます。

全体フレームワークは NIST SSDF のセキュア開発ライフサイクルの考え方、Google Testing の大規模検証体系、Kubernetes Production Readiness Review の機能本番投入準備メカニズム、ROS 2 REP-2004 のソフトウェアパッケージ品質宣言、Google SRE の運用信頼性管理を吸収しています——共通点：品質はリリース前チェックだけに頼れず、ライフサイクル全体を貫く必要があります。

## 2. 品質原則

8 項目。品質判断を専門に規律します：

1. **品質は証拠であり、感覚ではない** — システムが「利用可能」「安定」「安全」「完了」と呼ばれるには、検査可能な証拠が必要です。テスト結果、評価レポート、Review 記録、運用指標、セキュリティスキャン、model cards、シミュレーション記録、事故 postmortem はすべて証拠です。「問題ないはず」「先に上げて壊れたら直す」は証拠ではありません。

2. **リスクが品質強度を決める** — 短ライフサイクル、可逆、低影響変更は軽量検証；長期、クロスシステム、不可逆、セキュリティ / プライバシー / AI / 具身を伴う変更はより強い証拠が必要。証拠強度は `02-Security-Ethics.md` §3 の S0–S5 リスク分級と整合します。

3. **テスト左シフト、運用検証を欠かない** — 品質問題は設計、静的チェック、単体テスト、Review で可能な限り早期に露出すべきです。しかし、本番投入後のモニタリング、SLO、アラート、事故 postmortem、ユーザーフィードバックも品質証拠の一部であり、「運用の仕事」ではありません。

4. **小テスト多、大テスト少、クリティカルパスは end-to-end** — テスト構成は高速、確定的、特定可能な低層テストを基盤とし；統合テストで境界を検証；E2E はクリティカルユーザージャーニーと高リスク回帰パスのみをカバー。ピラミッド比率は調整可能ですが、逆転してはなりません。

5. **不安定テスト自体が欠陥** — 同一コードと設定で時に合格、時に失敗するテストは CI 信頼を破壊します。Flaky test は修正、隔離、または削除されなければなりません。繰り返し rerun で長期マスクすることは許可されません。

6. **品質は変更に内蔵し、リリース前に蓄積しない** — テスト、ドキュメント、可観測性、ロールバック、移行、セキュリティ検証は正常なエンジニアリング変更の一部であり、リリース前の一時的パッチではありません。

7. **品質問題はループを形成する必要がある** — Bug、回帰、事故、評価失敗、具身異常は根因まで追跡し、防回帰メカニズムを補完し、テスト、ドキュメント、モニタリング、設計制約に蓄積する必要があります。現象のみ修正し再発を防がないことは、修正していないことと同じです。

8. **AI と具身システムには追加品質次元が必要** — AI システムは従来ソフトウェアテストだけでは品質判断できません——振る舞い評価、対抗テスト、ドリフトモニタリング、権限レビューが必要です。具身システムはデジタルシミュレーションだけでは品質判断できません——物理境界、ハードウェアテスト、人間 takeover、E-Stop 検証が必要です。

## 3. 品質対象と品質等級

### 3.1 QA Level

輝夜計画は、異なるライフサイクルとリスクシナリオの産出物に 6 段階の品質等級を定義します。S0–S5 セキュリティ等級に代わるものではなく、各級に必要な**最低品質証拠強度**を規定します。

| 等級 | 名称 | 適用対象 | 最低品質証拠 |
|------|------|----------|-------------|
| **QA-L0** | Draft | 草稿、アイデア、非公式メモ | 状態の明記のみ、リリースコミットメントなし |
| **QA-L1** | Experiment | 研究 spike、一回限りスクリプト、ローカル実験 | 再現記録、データ出所説明、有効期限 |
| **QA-L2** | Maintained Component | 再利用可能ライブラリ、CLI、SDK、内部ツール | CI、単体テスト、ドキュメント、Owner、変更記録 |
| **QA-L3** | Production Service | ネットワークサービス、状態システム、データパイプライン、公共 API | 統合テスト、契約テスト、SLO、モニタリング、アラート、Runbook、ロールバック方案 |
| **QA-L4** | AI / Agent System | Agent、RAG、モデルサービス、長期記憶、ツール呼び出し | Eval Report、対抗テスト、データ/モデルカード、権限レビュー、振る舞い回帰、ドリフトモニタリング |
| **QA-L5** | Embodied / Safety-Critical | センサー、アクチュエータ、具身制御、高自由度物理システム | シミュレーション、SIL/HIL、物理境界、E-Stop 検証、HITL、事故 drill、独立安全レビュー |

QA-L と S 等級の関係：S 等級はセキュリティリスク高度を定義し、QA-L は品質証拠の広さを定義します。一般に S3 は QA-L3、S4 は QA-L4、S5 は QA-L5 に対応しますが、低セキュリティリスクの長期サービスも QA-L3 の運用品質証拠を維持すべきです。

### 3.2 品質等級宣言

QA-L2 以上のすべての正式リポジトリまたは長期コンポーネントは `QUALITY.md` を維持し、品質等級、カバレッジ範囲、バージョン戦略、テストカバレッジ、安全措置、運用保障、既知制限、レビュー周期を宣言する必要があります。この宣言はコンポーネントが依存者に対して行う公開品質コミットメントです——QA-L3 を主張するなら、証拠は QA-L3 の全要件を支える必要があります。

ROS 2 REP-2004 は同様の考え方を採用：ある品質等級を主張するパッケージは、バージョン戦略、変更制御、ドキュメント、テスト、依存管理の面でその等級の全条件を満たすことを証明する必要があります。輝夜計画はこの基盤の上に AI/Agent と具身システムの専門次元を追加しています。

## 4. 品質証拠体系

品質保障の核心はプロセスではなく証拠です。以下の分類は「何が証拠か」に答えます：

| 証拠種類 | 典型形式 | 適用段階 |
|----------|----------|----------|
| 設計証拠 | RFC、ADR、Engineering Brief、Threat Model、Hazard Analysis | 実装前 |
| 実装証拠 | PR、Code Review 記録、CODEOWNERS Approval | 開発期 |
| 自動化証拠 | CI レポート、テストレポート、Coverage、静的解析結果 | 継続 |
| セキュリティ証拠 | SAST、依存スキャン、Secret Scan、ASVS Checklist、コンテナスキャン | 継続 |
| サプライチェーン証拠 | SBOM、SPDX、SLSA Provenance、署名、イメージ digest | リリース時 |
| データ証拠 | Dataset Card、schema 検証、ドリフトレポート、重複除去レポート、出所宣言 | データ変更時 |
| モデル証拠 | Model Card、Eval Report、バージョン記録、Prompt/Policy snapshot | モデル変更時 |
| 運用証拠 | SLO レポート、モニタリング Dashboard、ログ、Tracing、アラート記録、事故 Postmortem | 運用期 |
| ユーザー証拠 | ユーザビリティテスト、可アクセシビリティレポート、フィードバック記録、Core Web Vitals | デリバリー後 |
| 具身証拠 | シミュレーション記録、SIL/HIL ログ、E-Stop テスト、物理テスト記録 | 具身システム全周期 |

原則：**証拠種類は品質等級と一致する必要があります**。QA-L2 に SLO レポートは不要；しかし QA-L3 にモニタリング証拠がなければ本番投入準備を主張できません。

## 5. 品質ゲート

品質ゲートは設計から運用までの連続チェックポイントであり、リリース前の一回性大チェックではありません。

### Gate Q0：Quality Planning

QA-L2 以上のすべての事項は、実装前に以下に答える必要があります：現在の品質等級、リスク等級、テスト戦略、必要な Review、必要な証拠、リリースゲート、ロールバック方法、責任者。これらの情報は Engineering Brief または Issue に記載します。

### Gate Q1：Design Quality

公共 API、長期状態システム、新インフラ、データパイプライン、Agent 振る舞い、モデルサービス、具身制御、ロールバック困難な移行に適用。レビューポイント：問題定義の明確性、不変量の定義、failure mode の特定、互換戦略の明確性、テストおよび観測計画の実行可能性、ロールバック戦略の存在。

Kubernetes KEP は各 enhancement proposal に Test Plan、Graduation Criteria、Production Readiness Review でこれらに事前回答することを要求します。輝夜計画の Design Quality Review は同一考え方の簡略版です。

### Gate Q2：Implementation Quality

PR 段階の品質チェック。コアロジックにテスト、エラー処理明確、新依存に出所とライセンス、公共契約変更同期、CI 合格、Review 完了。Google Code Review 標準と一致：変更はコードベース全体の健全性を継続的改善すべき。

### Gate Q3：CI Quality Gate

自動検証の硬いゲート。基礎項目：format、lint、typecheck、unit test、integration test、build、dependency scan、secret scan、license check。高リスク項目追加：契約テスト、E2E、移行 dry run、benchmark、セキュリティテスト、AI eval、データ検証、シミュレーションテスト、ロールバックテスト。CI 失敗はマージをブロックし、口頭承認は許可されません。

### Gate Q4：Release Quality Gate

リリース前最終確認。すべてのブロック欠陥クローズ、Release artifact 追跡可能、SBOM / provenance 生成、モニタリングとアラート就位、ロールバック方案利用可能、Owner が運用責任を受け入れ。高リスクリリースには Security Reviewer と AI/具身専門 Reviewer の署名も必要。

Google SRE Production Readiness Review と Kubernetes PRR は、機能が本番入り前に可観測、拡張可能、サポート可能、ロールバック可能であることを証明することを要求します。輝夜計画 Release Quality Gate はこの考え方の直接適用です。

### Gate Q5：Operational Quality Gate

上线後の継続品質検証。Health check、smoke test、error rate、latency、リソース用量、ログ、tracing、アラートノイズ、SLO burn rate、rollback readiness はすべて運用品質の証拠です。デプロイは品質の终点ではありません——運用フィードバックは証拠の最終環です。

## 6. テスト階層戦略

### 6.1 テストピラミッド

```text
Static Checks         ← 最速、最多、最廉価
  ↓
Unit Tests
  ↓
Component Tests
  ↓
Integration Tests
  ↓
Contract Tests
  ↓
E2E Tests
  ↓
Performance Tests
  ↓
Security Tests
  ↓
AI / Agent Evaluations
  ↓
Simulation / HIL
  ↓
Post-deploy Checks    ← 最遅、最少、最高価
```

原則：下層ほど多く、速く、確定的；上層ほど少なく、遅く、重要。輝夜計画は Google Testing Blog 経験比率（約 70/20/10）を初期参照として保持しますが、システム特性に応じ調整可能——前提はピラミッドを逆転しないこと。

### 6.2 Static Checks

テスト実行前に最低コストで低級エラーを排除：フォーマット、lint、型チェック、デッドコード検出、依存ポリシー、ライセンススキャン、secret スキャン、API schema 一貫性チェック。すべての正式リポジトリ CI に完全静的チェックを含める必要があります。

### 6.3 Unit Tests

純関数、ドメインロジック、状態遷移、シリアライズ、エラー処理、権限判断、データ変換を検証。要件：高速、確定的、実ネットワークまたは外部サービス非依存、個人ローカル環境非依存。テストが実データベースまたはモデルサービスを必要とするなら、単体テストではありません。

### 6.4 Component Tests

単一モジュールの境界振る舞いを検証：フロントエンドコンポーネント、バックエンド handler、Agent tool wrapper、データ validator、状態機コンポーネント。コンポーネントテストはモジュールインターフェース契約を検証し、内部実装詳細を重複テストしません。

### 6.5 Integration Tests

複数モジュール協業を検証：API + データベース、サービス + メッセージキュー、フロントエンド + mock バックエンド、Agent + tool sandbox、RAG + ベクトルストレージ、データパイプライン多段衔接、ROS ノード通信。統合テストは単体テストでは発見できない境界問題を露出します。

### 6.6 Contract Tests

輝夜計画はフロントエンド／バックエンド統一とドメイン契約統一を強調するため、契約テストは特殊な地位を持ちます。OpenAPI / gRPC / GraphQL schema、フロントエンド生成型、バックエンド実装一貫性、Event schema、状態機遷移ルール、Agent tool interface、ROS message/service/action に適用。

核心ルール：**公共契約変更には契約テストを伴う必要があります。破壊的変更には RFC、移行説明、バージョン戦略が必要です。**

### 6.7 E2E Tests

クリティカルユーザージャーニー、コアビジネスフロー、高リスク回帰パスのみをカバー。E2E は少なく重要——大量の脆い E2E で低層テストを代替しない。Playwright Trace Viewer は DOM snapshots、ネットワークリクエスト、スクリーンショットを記録でき、CI 内 E2E 失敗証拠アーカイブに適しています。

### 6.8 Performance Tests

API サービス、runtime scheduler、ベクトル検索、モデル推論、長コンテキスト処理、フロントエンドレンダリング、ロボット制御ループ、データパイプラインに適用。最低指標：latency p50/p95/p99、throughput、error rate、リソース消費、cold start、降格振る舞い。Google SRE ストレステスト原則が適用：システム将来信頼性への信心を定量化するのに役立ちます。

### 6.9 Security Tests

依存スキャン、SAST/CodeQL、認証認可テスト、入力検証、インジェクション防御、SSRF 防御、レート制限、コンテナスキャン。Web/API サービス最低参照 OWASP ASVS。

### 6.10 AI / Agent Evaluations

従来テストは AI 振る舞いをカバーできません——評価体系が必要です。§13 参照。

### 6.11 Simulation / HIL

具身システムの物理前検証。§14 参照。

## 7. 静的品質とコード健全性

### 7.1 静的チェックベースライン

すべての正式リポジトリは CI で以下を実行する必要があります：フォーマット、lint、型チェック、未使用依存検出、禁止依存ポリシー、生成ファイル一貫性、API schema diff、ライセンスメタデータ、secret スキャン。これらのチェック不合格はマージをブロックする必要があります。

### 7.2 コード健全性

コード健全性は「lint エラーなし」以上です。Review と継続保守は以下に注目：正確性、可読性、簡潔性、モジュール境界明確性、依存方向合理性、エラー処理完全性、並行安全、リソース解放、テスト可能性、可観測性、ドキュメント同期。

Google の Code Health 定義は方向として適切：コード理解を容易に、ライブラリを簡潔に、イテレーションを短く、安定性と性能を向上。輝夜計画 Code Review は単一コミットの完璧ではなく、長期コード健全性を目標とします。

## 8. Review 品質

`01-Workflow.md` は Review プロセスを定義；本節は Review が何をチェックすべきかを定義します。

### 8.1 Reviewer チェック次元

1. 変更が正しい問題を解決しているか；
2. 方案が既存アーキテクチャとドメインモデルに適合するか；
3. 複雑性が必要性によって証明されているか；
4. テストがコア振る舞いと failure path をカバーしているか；
5. 公共契約または互換性を破壊していないか；
6. セキュリティ、プライバシー、サプライチェーン、データリスクを導入していないか；
7. 運用、リリース、ロールバック、モニタリングに影響するか；
8. ドキュメントが同期されているか；
9. 先に RFC / ADR を経るべきか；
10. 追跡が必要な品質負債を導入しているか。

### 8.2 Review 結論

Review は「承認」と「拒否」のみではありません。結論は後続アクションを明確に導くべき：Approve、Request Changes、Needs RFC、Needs Security Review、Needs AI Eval、Needs Embodiment Safety Review、Needs Owner Decision。

### 8.3 高リスク変更の専門 Review

| 変更シナリオ | 追加 Review 要件 |
|--------------|------------------|
| 公共 API / Schema | API Owner + 影響クライアント代表 |
| データ処理 / プライバシー | Data Owner + Privacy Reviewer |
| モデル / Agent 振る舞い | AI Eval Reviewer + Security Reviewer |
| 具身制御 | Embodiment Safety Reviewer |
| 本番サービス | Infra / SRE Reviewer |
| サプライチェーン / リリースアーティファクト | Security / Release Reviewer |

通常 Review はコード健全性を保証；専門 Review は高リスク変更が専門的盲区に缺口を残さないことを保証します。

## 9. CI/CD 品質ゲート

CI は品質証拠の自動化生産ラインです。各リポジトリに自由发挥させるのではなく、workflow 構造を標準化することを推奨します。

### 9.1 基礎 CI

各正式リポジトリには以下を含める必要があります：メタデータ検証、依存ロックインストール、フォーマットチェック、lint、型チェック、単体テスト、統合テスト、ビルド、artifact アップロード。これらすべて合格して初めてマージ可能。

### 9.2 セキュリティ CI

独立セキュリティ workflow：secret スキャン、依存レビュー、SAST/CodeQL、ライセンスチェック、コンテナスキャン（該当時）、SBOM 生成。セキュリティ CI 失敗はエンジニアリング効率で上書きできません——セキュリティ証拠欠如は品質基準未達と同義です。

OWASP SAMM はセキュリティ能力を Governance、Design、Implementation、Verification、Operations に区分；輝夜計画セキュリティ CI は Implementation と Verification の自動検証部分をカバーします。

### 9.3 Release CI

リリース workflow：バージョン検証、Changelog チェック、artifact ビルド、SBOM と provenance 生成、署名、リリース候補、staging デプロイ、smoke test。Release CI は保護ブランチまたは tag からのみトリガー可能。

### 9.4 Nightly / Scheduled CI

低頻度だが深度の品質検証：低速テスト、全量回帰、性能 benchmark、AI eval suite、データドリフトチェック、依存更新 dry run、flaky test 検出。PR 級トリガーには遅すぎますが、全体品質ベースラインに不可欠です。

## 10. API、プロトコル、契約品質

輝夜計画はフロントエンド、バックエンド、Agent、Infra、データ、具身実行を跨ぐため、契約一貫性はクロスシステム信頼性の基盤です。

### 10.1 契約品質要件

- Schema に権威ソース（OpenAPI / Proto / GraphQL SDL）；
- API にバージョン戦略と弃用ウィンドウ；
- エラー形式がサービス間で一貫；
- Breaking change に自動検出と RFC プロセス；
- フロントエンド型は契約から生成、手書き不可；
- Event / Message に互換性検証；
- 状態機遷移ルールに形式化テスト；
- Deprecated フィールドに明確な移行タイムライン。

### 10.2 契約テスト方式

推奨組合せ：schema diff 自動比較、生成クライアントコンパイルテスト、consumer-driven contract test、後方互換テスト、移行 dry run、golden file snapshot、conformance suite。

Kubernetes Conformance は異なる実装が相互運用のため必要 API をサポートすることを保証。輝夜計画は公共プロトコルと Agent Tool API 向けに独自 conformance suite を確立し、複数リポジトリ、複数言語実装間の静かなドリフトを防げます。

---

## 11. フロントエンド、ユーザビリティ、可アクセシビリティ品質

フロントエンド品質は「ページが開ける」以上です。

### 11.1 フロントエンド品質次元

機能正確性、コンポーネント一貫性、状態管理正確性、API 契約一致、可アクセシビリティ（WCAG）、性能（Core Web Vitals）、レスポンシブレイアウト、エラー状態処理、空状態と Loading 状態、国際化準備、ビジュアル回帰。

### 11.2 フロントエンドテスト要件

コンポーネントテストで独立コンポーネント振る舞いをカバー；API mock / contract test でフロントエンド／バックエンド契約一貫性を保証；E2E でコアインタラクションパスをカバー；可アクセシビリティチェックでキーボードナビゲーションと支援技術互換を保証；性能予算で LCP、INP、CLS が上限内であることを保証。

Core Web Vitals は LCP ≤ 2.5s、INP ≤ 200ms、CLS ≤ 0.1 を良好体験閾値とします。輝夜計画フロントエンドプロジェクトはこれを性能ベースラインとします。

### 11.3 ユーザビリティ検証

クリティカル界面にはユーザビリティ検証を実施すべきです。特に Agent 状態可視化、評価 Dashboard、シミュレーション制御パネル、Onboarding フロー。ユーザビリティテストの核心は、実ユーザーがタスク実行時の振る舞いと障害を観察すること——ISO 9241-11 はユーザビリティを、特定ユーザーが特定環境で効果性、効率、満足度をもって目標達成する程度と定義します。

## 12. バックエンド、サービス、運用品質

### 12.1 サービス品質次元

API 正確性、データ一貫性、トランザクション境界、並行安全、タイムアウトとリトライ戦略、冪等性、認証認可、レート制限、リソース隔離、可観測性、ロールバックと降格、バックアップ復旧、移行安全。

### 12.2 バックエンドテスト要件

単体テストでビジネスロジックをカバー；統合テストでデータベース相互作用と外部サービス呼び出しをカバー；契約テストで API 一貫性を保証；移行テストで schema 変更安全を検証；認可テストで権限境界を検証；故障注入で降格振る舞いを検証；負荷テストで容量を検証。

### 12.3 SLO と運用品質

QA-L3 以上のすべてのサービスは SLO を定義し、少なくとも可用性、latency、error rate をカバーする必要があります。データパイプラインには freshness SLO も；ストレージシステムには durability SLO も必要。各 SLO には対応する error budget policy が必要——予算耗尽時のアクションは空白にできません。

Google SRE の核心洞察：100% 信頼性は現実的でも理想的でもありません。SLO と error budget の価値は、信頼性とイノベーション速度の間に明確で actionable な平衡点を確立すること。輝夜計画は事故後に临时 SLO を補うのではなく、この体系を段階的に確立すべきです。

## 13. データ、モデル、AI Agent 品質

輝夜計画を通常ソフトウェアプロジェクトと区別する重要章です。従来テスト体系は AI システム品質をカバーできません——評価、対抗テスト、ドリフトモニタリング、振る舞いレビューが必要です。

### 13.1 データ品質

すべての正式データセット（訓練集、評価集、RAG ドキュメントライブラリ、長期記憶データ）の品質ゲート：

- **出所とライセンス**：provenance 明確、license コンプライアンス、未授权保護表現なし；
- **構造完全**：schema 固定、型正確、欠損率制御、重複汚染なし；
- **分布健全**：異常ドリフトなし、train/eval 漏洩なし、機微フィールドのラベル付けまたは脱敏済み；
- **バージョン制御**：データセットにバージョン、変更記録、保留ポリシー。

TensorFlow Data Validation の schema 推論とドリフト検出考え方は参考実装として使用可能。Datasheets for Datasets が提案するデータ説明ドキュメント（動機、構成、収集プロセス、推奨用途）は輝夜計画 Dataset Card の最低内容とすべきです。

### 13.2 モデル品質

モデルまたはモデルサービスの品質証拠：適用シナリオ、範囲外使用、訓練データ出所、評価データと指標、サブグループ性能差、既知制限、ロバスト性、キャリブレーション品質、推論 latency とコスト、ドリフトトレンド、安全振る舞い、降格戦略。

Model Cards は公開モデルに適用境界、評価プロセス、局限の説明ドキュメントを添付することを要求。輝夜計画の各正式モデルサービスは Model Card を維持すべきです。

### 13.3 AI 本番投入準備チェック

Google ML Test Score 考え方を参考に、輝夜計画は AI 本番投入準備の 4 次元を定義：

**データ次元**：schema 固定、訓練/評価セットバージョン固定、ドリフトモニタリング整備、漏洩検出完了、出所とライセンス明確。

**モデル次元**：バージョン固定、評価結果アーカイブ、性能ベースライン存在、回帰 suite 存在、fallback 戦略存在。

**システム次元**：推論 latency モニタリング、コスト予算モニタリング、エラー処理完备、入出力検証、リリースとロールバックメカニズム整備。

**モニタリング次元**：オンライン品質指標、ドリフト検出、幻覚/不確実性代理指標、ユーザーフィードバックループ、事故報告メカニズム。

4 次元すべて達成して初めて、AI システムは本番投入準備を主張できます。

### 13.4 Agent 品質

Agent はモデルだけではありません——ツール、記憶、ポリシー、権限境界を持つ振る舞いシステムです。Agent 品質は以下をカバーする必要があります：

- System prompt / policy バージョン制御；
- ツール権限マトリクスレビュー；
- ツール呼び出し正確性と拒否テスト；
- Prompt injection 対抗回帰；
- 記憶読み書き振る舞いテストと記憶 poisoning 防御；
- RAG 検索精度と recall；
- 長コンテキスト安定性；
- 人格/振る舞い一貫性；
- 不確実性処理と拒否振る舞い；
- 危険動作ブロック；
- コスト/latency 予算；
- ループ検出と人間 handoff；
- 監査ログ完全性。

OWASP LLM Top 10 は Prompt Injection、Insecure Output Handling、Training Data Poisoning などを重要リスクとして列挙。輝夜計画 Agent 品質ゲートはこれら既知攻撃面をカバーする必要があります。

### 13.5 AI Eval Report

AI/Agent システムの各正式リリースは評価レポートを生成し、以下を記録：システムバージョン、評価範囲、評価データ出所とバージョン、タスク種類、コア指標、ベースラインとの比較、回帰状況、セキュリティテスト結果、人工レビュー結論、最終判定（Pass / Conditional Pass / Fail）、既知制限、後続フォローアップ Issue。

## 14. 具身知能と物理安全品質

具身システムのエラーは物理的結果を生みます——品質要件は純ソフトウェアより厳格である必要があります。

### 14.1 具身品質段階

```text
Design Hazard Analysis
  ↓
Simulation
  ↓
Software-in-the-loop (SIL)
  ↓
Hardware-in-the-loop (HIL)
  ↓
Controlled Physical Test
  ↓
Human-in-the-loop Operation
  ↓
Limited Autonomy
  ↓
Operational Acceptance
```

各ステップは次に進む前に品質証拠が必要です。ステップ飛ばしは具身安全で最も一般的な失敗モードです。

### 14.2 具身検証要件

- 危害分析がすべての特定リスクシナリオをカバー；
- センサー有効性と故障応答が検証済み；
- アクチュエator 速度、力、トルク制限が設定されテスト済み；
- 作業空間境界が定義され越え不可；
- 衝突処理戦略が検証済み；
- 通信タイムアウト時にシステムが安全状態に入る；
- E-Stop がすべてのシナリオで有効；
- 人間が任意の時点で takeover 可能；
- すべての物理テストに完全ログアーカイブ。

NASA ソフトウェア保障標準は、安全クリティカルソフトウェアが危険を招く乱序コマンドを拒否し、メモリ改ざんを検出して安全状態に復帰し、入出力に完全性チェックを行うことを要求。ISO/TS 15066 は協働ロボットシステムの安全要件を規定。輝夜計画具身システムは設計段階でこれら制約を内蔵し、統合後に補完してはなりません。

### 14.3 ROS / ロボットミドルウェア品質

ROS 2 または類似ミドルウェア使用時、コアパッケージとメッセージインターフェースは Quality Declaration（REP-2004 参照）を維持、QoS profile 定義、メッセージ互換性検証、lifecycle state 遷移テスト、故障注入、latency 予算と安全境界検証を実施すべきです。

## 15. セキュリティとサプライチェーン品質

本節は `02-Security-Ethics.md` と重複しますが、ここでは**検証証拠**に焦点——セキュリティポリシーは安全文書に、セキュリティ証拠は品質保障に属します。

### 15.1 セキュリティ検証要件

すべての正式リポジトリは以下を実行する必要があります：secret スキャン、依存スキャン、依存 review、ライセンスチェック、SAST。リリースアーティファクトにはさらに：コンテナスキャン、SBOM 生成、provenance 記録、artifact 署名。

OpenSSF Scorecard はオープンソースプロジェクトのセキュリティヒューリスティック指標を自動チェック可能。SLSA は provenance でビルド主体とプロセスを記録。Sigstore はリリースアーティファクトの署名と検証に使用可能。輝夜計画正式リリースは段階的に SLSA Level 2 を達成し、クリティカルシステムは Level 3 を目指します。

### 15.2 Web / API セキュリティ検証

Web と API サービス最低参照 OWASP ASVS。認証、認可、セッション、入力検証、出力エンコーディング、インジェクション防御、ファイルアップロード、ログ、エラー処理、レート制限をカバー。セキュリティテストは任意追加ではありません——QA-L3 以上品質等級の必要証拠です。

## 16. リリース品質と本番投入準備

### 16.1 Release Quality Report

各正式リリース（Draft / 実験的リリース以外）は Release Quality Report を生成し、以下を記録：バージョンとビルド情報、テスト要約（各層合格状況）、オープン欠陥、リスク評価と受容リスク、運用就绪状態（モニタリング/アラート/Runbook/ロールバック）、最終判定（Go / No-Go / Conditional Go）、承認記録。

### 16.2 本番投入準備チェック

QA-L3 サービス本番投入準備最低要件：CI 全合格、統合と契約テスト合格、データ移行 dry run 合格（該当時）、モニタリングとアラート設定済み、SLO 定義済み、Runbook 完了、ロールバック方案検証済み、Backup Owner 確認済み、セキュリティスキャン合格。

QA-L4 AI/Agent サービス追加：モデル/prompt/policy バージョン固定、Eval Report 完了、Prompt Injection テスト合格、ツール権限マトリクスレビュー完了、振る舞い回帰合格、ロールバック/無効化メカニズム利用可能。

QA-L5 具身システム追加：Hazard analysis 完了、シミュレーションと SIL/HIL テスト合格、物理境界定義完了、センサー故障テスト合格、HITL 利用可能、E-Stop 検証合格、独立安全 Reviewer 承認。

Kubernetes Release Cycle の Code Freeze とドキュメントレビュー段階は品質管理の良い例——機能は最終リリース前に临时補完するのではなく、事前にテスト計画と本番投入準備レビューを完了する必要があります。輝夜計画正式バージョンリリースはこの凍結とチェックメカニズムを参考にすべきです。

## 17. 欠陥、回帰、事故ループ

### 17.1 欠陥ライフサイクル

```text
Report → Triage → Reproduce → Classify → Root Cause → Fix → Regression Test → Release → Verify → Close
```

各環に Owner があり、欠陥がいずれかの環で無期限滞留することは許可されません。

### 17.2 欠陥分級

| 等級 | 意味 | 応答要件 |
|------|------|----------|
| **P0** | 本番中断、セキュリティ漏洩、具身危険、データ破損 | 即時対応 |
| **P1** | クリティカル機能不可用、深刻回帰、重大プライバシーリスク | 優先修正 |
| **P2** | 重要機能異常、workaround 存在 | 通常イテレーション修正 |
| **P3** | 小 bug、ドキュメント問題、低影響体験問題 | 保守計画内対応 |
| **P4** | 改善提案、クリーンアップ項目 | Backlog |

### 17.3 修正は品質証拠を補償する必要がある

Bug 修正は「変更完了」ではありません。各修正は少なくとも以下の 1 つを追加：回帰テスト、より強い検証、より明確なエラーメッセージ、より完善なログ、ドキュメント更新、モニタリングアラート、Runbook 項目。Bug を修正し防御線を補わないことは、同じ場所で次の転倒を待つことと同じです。

### 17.4 Flaky Test ガバナンス

Flaky test 処理フロー：検出 → マーキング → Owner 割当 → 隔離（必要時のみ）→ 修正または削除 → 再有効化 → 再発追跡。

ルール：Flaky test は main を長期ブロックしてはならない；隔離には Owner と修正 SLA が必要；习惯性 rerun で不安定性をマスクすることは許可されない；高リスクパス上の flaky test は優先修正。

## 18. 品質メトリクス

品質指標は体系的問題発見、プロセス改善、リスク低減のためであり、個人の機械的排名や貢献者への罰則のためではありません。

### 18.1 推奨指標

| カテゴリ | 主要指標 |
|----------|----------|
| コード品質 | lint/typecheck failure rate、review defect density、complexity hotspots |
| テスト品質 | test pass rate、flaky test rate、coverage trend |
| 欠陥品質 | escaped defect rate、regression rate、P0/P1 count、time to fix |
| デリバリー品質 | lead time、deployment frequency、change failure rate、recovery time |
| 運用品質 | availability、latency、error rate、SLO burn、incident count |
| セキュリティ品質 | open critical vulnerabilities、secret leaks、dependency age、SBOM coverage |
| データ品質 | schema violation rate、drift events、contamination incidents |
| AI 品質 | eval pass rate、hallucination proxy、unsafe output rate、tool-call failure rate |
| 具身品質 | simulation pass rate、near-miss count、E-Stop test pass rate |
| ユーザー品質 | task success rate、accessibility issues、Core Web Vitals、support issue volume |

DORA は deployment frequency、lead time、change failure rate、time to restore をデリバリーと安定性の核心指標とします。輝夜計画は QA-L3+ すべてのサービスにこの 4 指標の可観測性を段階的に確立すべきです。

## 19. 品質例外と品質負債

例外は許可する必要があります——しかし例外は無料ではありません。

### 19.1 Quality Exception

品質例外条件：Owner あり、理由あり、影響範囲あり、临时緩和措置あり、有効期限あり、再審日期あり。セキュリティ、プライバシー、法律、具身安全、未授权資産底线を突破してはならない。各例外には対応する追跡 Issue が必要。

### 19.2 Quality Debt

品質負債：テスト欠如、ドキュメント欠如、モニタリング欠如、期限切れ依存、flaky test、手動デプロイ、ロールバック方案なし、Owner なし、モデル評価不足、データ provenance 不完全、具身シミュレーションカバレッジ不足。

ルール：品質負債は PR コメントと個人記憶にのみ存在してはならない——Issue / Project に入り追跡可能な作業項目となる必要があります。高リスク品質負債（セキュリティ、本番、具身安全に影響）は修正 SLA が必要。

## 20. ツール、自動化、Agent 支援

### 20.1 Kaguya QA Sentinel

輝夜計画は QA Sentinel Agent をデプロイして品質保障工作を支援できます：CI 失敗トレンド集約、flaky tests マーキング、Release Quality Report 草稿生成、PR のテスト説明または契約テスト欠如チェック、AI 変更の Eval Report 欠如チェック、データ変更の Dataset Card 欠如チェック、具身変更のシミュレーションログ欠如チェック、週次 Quality Digest 生成。

> QA Sentinel は通知、集約、チェック、レポート生成ができます。PR 自動承認、CI 迂回、失敗テスト削除、専門 Reviewer 判断の代替はしてはなりません。Agent は品質を支援し、品質を裁定しません。

### 20.2 自動化戦略

- 自動化可能なすべてのチェックを CI で実行し、人工 Review の低級エラー負担を減らす；
- 契約チェック、schema diff、依存レビューは自動でコメントまたはラベルをトリガー；
- 性能回帰と AI eval 回帰は自動で比較レポートを生成し PR に添付；
- 品質指標は定期的に自動収集・可視化し、人工統計に頼らない；
- アラートノイズと flaky test トレンドは自動追跡し Owner に通知。

## 21. テンプレートとチェックリスト

以下テンプレートは輝夜計画品質保障体系の配套産出物です。具体テンプレート内容はエンジニアリング実践と共に迭代し、`templates/` または `05-Knowledge/` に置けます：

- `QUALITY.md` — 品質等級宣言
- Test Plan — テスト計画
- Release Quality Report — リリース品質レポート
- AI Evaluation Report — AI 評価レポート
- Dataset Card — データセット説明
- Model Card — モデル説明
- Embodiment Test Report — 具身テストレポート
- Flaky Test Report — 不安定テストレポート
- Defect Report — 欠陥レポート
- Quality Exception — 品質例外申請
- Post-release Review — リリース後振り返り

## 22. アンチパターン

以下は品質保障アンチパターンです：

1. QA を「リリース前にテストを 1 回回す」と理解する；
2. E2E のみで、単体、統合、契約テストなし（逆ピラミッド）；
3. flaky test を修正せず繰り返し rerun でマスク；
4. CI 失敗を口頭承認またはマージ迂回；
5. 品質ゲートなしプロトタイプが直接本番依存に入る；
6. 公共 API 変更に契約テストと互換性検証なし；
7. データセットに provenance、license、schema チェックなし；
8. モデルを benchmark スコアのみで判断し、failure mode と安全振る舞いをテストしない；
9. Agent が新ツール権限を得るが権限マトリクステストなし；
10. RAG システムに検索品質検証と汚染検出なし；
11. 具身システムがシミュレーションのみで物理動作を直接開放；
12. Bug 修正に回帰テストを補わない；
13. rollback plan と運用就绪確認なしでリリース；
14. 本番投入後にモニタリングを見ず、アラートを設定せず、Runbook を書かない；
15. 品質負債が口頭約定と PR コメントにのみ存在；
16. テストカバレッジを品質自体とみなす（高カバレッジ ≠ 高品質）；
17. 速度追求のため品質ベースラインを長期低下させ代价を追跡しない；
18. AI 生成テストを使用するが誰も振る舞いを理解せず正確性を検証しない。

## 23. 改訂

本文書は公開 RFC による改訂のみ可能です。改訂には、品質要件が過重または過軽と証明されたか、新種類システムにカバレッジが欠けるか、あるルールが実践で工程健全性を保護ではなく阻害すると証明されたかを明記する必要があります。`01-Foundation/01-Principles.md` の「衝突と改訂」と一致：本文書がセキュリティレビューまたは組織権限と衝突する場合、対応する専門文書が優先；法律、セキュリティ倫理ボトムラインと衝突する場合、ボトムラインが優先。旧版はバージョン管理に保存され、いつでも参照可能。

本文書の硬い核心は 5 項目のみ：

1. 各正式資産は品質等級を宣言する必要がある。
2. 各品質等級には最低証拠要件がある。
3. 各高リスク変更は対応する品質ゲートを通過する必要がある。
4. 各欠陥と事故は防回帰メカニズムとして沉淀する必要がある。
5. 各 AI / Agent / 具身システムには従来ソフトウェアテスト以外の专项品質証拠がある。

この 5 項目を守ることで、輝夜計画は実験と Demo から長期保守可能、リリース可能、監査可能、運用可能なエンジニアリング体系へ安定移行できます。
