# エンジニアリングと開発

ソフトウェアエンジニアリングの実践、バージョン管理、ブランチ戦略、開発ワークフローに関する用語集です。輝夜計画の開発標準の基礎となる概念を定義します。

---

## GitHub Flow

**日本語名：** GitHub Flow

短命のフィーチャーブランチですべての変更を開発し、Pull Requestを通じて`main`にマージする軽量なブランチベースのワークフロー。Git Flowよりシンプルで、継続的デリバリーに適している。輝夜計画では標準的な開発フローとして採用している。

## Git Flow

**日本語名：** Git Flow

長寿命の`develop`、`release`、`hotfix`ブランチを持つより重いブランチモデル。継続的デリバリーのワークフローには不要な複雑さがあるため、輝夜計画では採用していない。

## Pull Request (PR)

**日本語名：** プルリクエスト（PR）

コードベースへの変更を提案する仕組み。PRには差分、説明、テスト計画、メタデータが含まれ、保護されたブランチにマージされる前にCIチェックと人間によるレビューを通過する必要がある。

## Draft PR

**日本語名：** ドラフトPR

作業中としてマークされ、マージできないPR。実装方針を早期に公開し、予備的なフィードバックを得て、正式レビュー前にCIの問題を検出するために使用する。

## Branch Protection

**日本語名：** ブランチ保護

保護されたブランチ（通常`main`）にルールを適用するGitHubリポジトリ設定。PRレビュー、ステータスチェックの通過、CODEOWNER承認、リニア履歴の要求、force pushの禁止などを含む。

## Conventional Commits

**日本語名：** 規約に基づくコミット（Conventional Commits）

`<type>(<scope>): <description>`形式のコミットメッセージ規約。`feat`、`fix`、`docs`、`refactor`、`test`、`perf`、`ci`、`chore`などのタイプを使用する。SemVerと整合し、自動化された変更履歴とバージョン管理を可能にする。

## SemVer (Semantic Versioning)

**日本語名：** セマンティックバージョニング（SemVer）

`MAJOR.MINOR.PATCH`形式のバージョニング方式。MAJORは破壊的変更、MINORは後方互換の機能追加、PATCHは後方互換の修正を示す。公開APIを持つコンポーネントに適用される。

## CI/CD (Continuous Integration / Continuous Delivery)

**日本語名：** 継続的インテグレーション／継続的デリバリー（CI/CD）

CIはすべてのコード変更を自動的にビルド、テスト、検証する。CDは検証済みの変更をステージングまたは本番環境に自動デプロイする。両者を合わせて、コミットからデプロイまでの自動化パイプラインを形成する。

## GitHub Actions

**日本語名：** GitHub Actions

ワークフローをYAMLファイルで定義するGitHub組み込みのCI/CDプラットフォーム。マトリクスビルド、環境シークレット、アーティファクトのアップロード、デプロイ保護ルールをサポートする。輝夜計画のCI/CD基盤として利用している。

## Lockfile

**日本語名：** ロックファイル

正確な依存関係バージョンとその推移的依存関係を固定するファイル（例：`uv.lock`、`pnpm-lock.yaml`、`Cargo.lock`）。環境とCI間で再現可能なビルドを保証する。

## Dev Container

**日本語名：** 開発コンテナ（Dev Container）

`devcontainer.json`で定義されるコンテナ化された開発環境。GPU依存関係、システムライブラリ、ROS、シミュレータなど複雑なセットアップに対して、再現可能で事前設定された環境を提供する。

## Feature Flag

**日本語名：** フィーチャーフラグ

新しいコードをデプロイすることなく、実行時に機能を有効または無効にするトグル。段階的ロールアウト、A/Bテスト、機能の即時ロールバックを可能にする。

## Kill Switch

**日本語名：** キルスイッチ

完全なデプロイサイクルなしに、本番環境で機能、サービス、または動作を即座に無効化する緊急メカニズム。高リスクシステムにとって不可欠である。輝夜計画のAIエージェントや身体性システムでも必須の安全装置として位置づけられる。

## Hotfix

**日本語名：** ホットフィックス

重大な本番問題に対する迅速な修正。短縮されたレビュープロセスに従うが、記録、テスト、根本原因分析、回帰テストの追加は依然として必要である。

## Squash Merge

**日本語名：** スカッシュマージ

PR内のすべてのコミットをターゲットブランチ上の単一コミットに統合するマージ戦略。個々のコミット粒度を犠牲にして、よりクリーンなリニア履歴を生成する。

## Linear History

**日本語名：** リニア履歴

スカッシュマージまたはリベースによってマージコミットのないブランチ履歴を実現する。複雑なマージトポロジーを持つ履歴よりも読みやすく、bisectや原因分析が容易である。

## Matrix Testing

**日本語名：** マトリクステスト

OS、言語バージョン、ランタイム、または設定の複数の組み合わせで同じテストスイートを実行する。CIワークフローで定義し、環境固有の失敗を検出する。

## Monorepo

**日本語名：** モノレポ

複数のプロジェクト、パッケージ、またはサービスを含む単一リポジトリ。プロジェクト横断の変更を簡素化するが、選択的ビルドとテストのためのツールが必要である。

## Workspace

**日本語名：** ワークスペース

単一リポジトリ内の複数パッケージを管理するパッケージマネージャー機能（pnpm workspace、Cargo workspace）。依存関係を共有し、パッケージ間の開発を可能にする。

## Engineering Brief

**日本語名：** エンジニアリングブリーフ

RFCより短く、Issueより詳細な軽量設計文書。標準的または高リスクのエンジニアリング変更を実装する前に作成する。問題、方針、代替案、リスク、テスト計画、オーナーシップを記録する。輝夜計画ではRFCとIssueの中間的な設計文書として位置づけられる。
