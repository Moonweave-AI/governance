# 品質とテスト

ソフトウェア品質保証、テスト戦略、検証の実践に関する用語集です。輝夜計画のQAレベル（QA-L0〜L5）と品質ゲートの基礎概念を定義します。

---

## Test Pyramid

**日本語名：** テストピラミッド

テスト分布のモデル。底辺に多数の高速で独立したユニットテスト、中間に少ない統合テスト、頂点に最小限の低速E2Eテストを配置する。おおよその比率は70%ユニット／20%統合／10%E2Eである。

## Unit Test

**日本語名：** ユニットテスト

単一の関数、メソッド、またはクラスを分離して検証するテスト。高速で決定的であり、ネットワーク、データベース、外部サービスに依存してはならない。

## Component Test

**日本語名：** コンポーネントテスト

単一モジュールの境界動作を検証するテスト（例：UIコンポーネント、APIハンドラー、ステートマシン）。内部実装の詳細はテストしない。

## Integration Test

**日本語名：** 統合テスト

複数のモジュールまたはサービス間の相互作用を検証するテスト（例：API＋データベース、サービス＋メッセージキュー、Agent＋ツールサンドボックス）。輝夜計画ではエージェントと外部ツールの連携検証に特に重要である。

## Contract Test

**日本語名：** コントラクトテスト

共有インターフェース仕様（APIスキーマ、イベント形式、ステートマシン遷移）への準拠を検証するテスト。プロバイダーとコンシューマーの互換性を維持する。

## Consumer-Driven Contract Test

**日本語名：** コンシューマー主導コントラクトテスト

コンシューマーが期待値を定義し、プロバイダーがすべてのコンシューマーコントラクトを満たすことを検証するコントラクトテスト手法。破壊的変更が依存先に到達するのを防ぐ。

## E2E Test (End-to-End Test)

**日本語名：** エンドツーエンドテスト（E2Eテスト）

ブラウザまたはAPIクライアントを通じて、ユーザーの視点からシステム全体を実行するテスト。重要なユーザージャーニーをカバーするが、低速で高コスト、不安定になりうる。

## Regression Test

**日本語名：** 回帰テスト

以前修正したバグの再発を防ぐために追加されるテスト。すべてのバグ修正には少なくとも1つの回帰テストを含めるべきである。

## Flaky Test

**日本語名：** 不安定テスト（Flaky Test）

同じコードと設定で一貫しない結果（合格／不合格）を返すテスト。CIの信頼性を損ない、修正、隔離、または削除が必要である。

## Static Analysis

**日本語名：** 静的解析

実行なしでコードを自動検査する。フォーマット、リント、型チェック、デッドコード検出、依存関係ポリシーの適用、セキュリティパターンのスキャンを含む。

## Code Coverage

**日本語名：** コードカバレッジ

テストによって実行されたコード行、分岐、またはパスの割合。有用なトレンド指標であるが品質の保証ではない。高いカバレッジは正確性を保証しない。

## Mutation Testing

**日本語名：** ミューテーションテスト

コードに小さな変更（ミューテーション）を導入し、テストがそれを検出するかを確認する手法。単純なカバレッジ指標を超えたテストスイートの有効性を測定する。

## QA Level (QA-L0 through QA-L5)

**日本語名：** 品質レベル（QA-L0〜L5）

輝夜計画の品質エビデンススケール。L0（ドラフト）、L1（実験）、L2（保守コンポーネント）、L3（本番サービス）、L4（AI/Agentシステム）、L5（身体性／安全クリティカル）。各レベルには段階的に強い品質エビデンスが要求される。

## Quality Gate

**日本語名：** 品質ゲート

作業が前進する前に通過しなければならないチェックポイント。例：Gate Q0（品質計画）、Q1（設計レビュー）、Q2（実装レビュー）、Q3（CIゲート）、Q4（リリースゲート）、Q5（運用ゲート）。

## Conformance Test

**日本語名：** 適合性テスト

実装が仕様を正しくサポートすることを検証するテストスイート。公開プロトコル、API、実装間の相互運用性に使用される。

## Golden File Test

**日本語名：** ゴールデンファイルテスト

出力を既知の正しいスナップショットと比較するテスト。生成出力、シリアライゼーション、APIレスポンスの意図しない変更の検出に有用である。

## Performance Test

**日本語名：** パフォーマンステスト

さまざまな負荷条件下でのレイテンシ、スループット、リソース消費、劣化動作を測定するテスト。

## Load Test

**日本語名：** 負荷テスト

想定される本番トラフィック量をシミュレートし、システムが通常負荷を劣化なく処理できることを検証するパフォーマンステスト。

## Stress Test

**日本語名：** ストレステスト

想定限界を超えてシステムに負荷をかけ、破綻点、劣化動作、回復特性を特定するパフォーマンステスト。

## Smoke Test

**日本語名：** スモークテスト

デプロイ直後に実行する最小限のテストセット。システムが基本的に動作していることを検証し、致命的なデプロイ失敗を迅速に検出する。

## SLO (Service Level Objective)

**日本語名：** サービスレベル目標（SLO）

サービスの信頼性に対する目標。パーセンテージまたは閾値で表現される（例：「99.9%のリクエストが200ms以内に完了」）。サービスにとって「十分に信頼できる」とは何を意味するかを定義する。

## Error Budget

**日本語名：** エラーバジェット

SLOから導出される許容可能な非信頼性の量。SLOが99.9%の場合、エラーバジェットは0.1%である。予算が使い果たされると、新機能より信頼性改善が優先される。
