# 质量与测试

本文档定义 Moonweave/辉夜计划（Kaguya Project）治理文档中使用的软件质量保证、测试策略与验证实践相关术语。

---

## Test Pyramid

**中文名称：** 测试金字塔

测试分布模型：底层大量快速、隔离的单元测试；中层较少集成测试；顶层最少慢速 E2E 测试。近似比例约为 70% 单元 / 20% 集成 / 10% E2E。辉夜计划各组件的测试策略应与此金字塔对齐。

## Unit Test

**中文名称：** 单元测试

验证单个函数、方法或类行为的测试。须快速、确定且独立于网络、数据库或外部服务。构成测试金字塔的基座，是 CI 每次提交必跑的核心测试层。

## Component Test

**中文名称：** 组件测试

验证单个模块边界行为的测试（如 UI 组件、API 处理器或状态机），不测试内部实现细节。介于单元测试与集成测试之间，适用于辉夜计划 Agent 模块与 API 边界验证。

## Integration Test

**中文名称：** 集成测试

验证多个模块或服务之间交互的测试（如 API + 数据库、服务 + 消息队列、Agent + 工具沙箱）。捕获接口契约与协作逻辑问题，是 QA-L2 及以上组件的必备证据。

## Contract Test

**中文名称：** 契约测试

验证对共享接口规范（API schema、事件格式、状态机转换）遵守情况的测试。确保提供方与消费者保持兼容，对辉夜计划跨仓库与跨服务集成尤为重要。

## Consumer-Driven Contract Test

**中文名称：** 消费者驱动契约测试

契约测试方法：由消费者定义期望，提供方验证满足所有消费者契约。防止破坏性变更波及依赖方，适用于 Agent 工具接口与内部 API 演进。

## E2E Test (End-to-End Test)

**中文名称：** 端到端测试（E2E）

从用户视角演练完整系统的测试，通常通过浏览器或 API 客户端执行。覆盖关键用户旅程，但慢、成本高且可能不稳定。辉夜计划将其控制在测试金字塔顶端少量关键路径。

## Regression Test

**中文名称：** 回归测试

专门用于防止已修复缺陷再次出现的测试。每次缺陷修复应至少附带一条回归测试；是辉夜计划质量门禁与 Postmortem 跟进动作的常规要求。

## Flaky Test

**中文名称：** 不稳定测试

在相同代码与配置下产生不一致结果（时过时不时）的测试。会侵蚀 CI 可信度，须修复、隔离或移除。辉夜计划 CI 政策不允许长期容忍 Flaky Test。

## Static Analysis

**中文名称：** 静态分析

不执行代码的自动化代码检查：格式化、lint、类型检查、死代码检测、依赖策略执行与安全模式扫描。与单元测试互补，是 PR 合并前的常规门禁。

## Code Coverage

**中文名称：** 代码覆盖率

测试所覆盖的代码行、分支或路径的百分比。是有用的趋势指标，但不是质量保证——高覆盖率不意味着正确性。辉夜计划将其作为 QA Level 证据之一，而非唯一标准。

## Mutation Testing

**中文名称：** 变异测试

向代码引入微小变更（变异）并检查测试能否发现的技术。衡量测试套件有效性，超越简单覆盖率指标。适用于 QA-L3 及以上关键组件的深度质量评估。

## QA Level (QA-L0 through QA-L5)

**中文名称：** 质量等级（QA-L0～L5）

辉夜计划的质量证据分级：L0（草稿）、L1（实验）、L2（维护组件）、L3（生产服务）、L4（AI/Agent 系统）、L5（具身/安全关键）。每一级要求递进更强的质量证据，与 Moonweave 成熟度 M 等级协同。

## Quality Gate

**中文名称：** 质量门禁

工作推进前必须通过的检查点。示例：Gate Q0（质量规划）、Q1（设计审查）、Q2（实现审查）、Q3（CI 门禁）、Q4（发布门禁）、Q5（运维门禁）。与项目 Gate 及 Release Readiness 流程衔接。

## Conformance Test

**中文名称：** 一致性测试

验证实现是否正确支持某规范的测试套件。用于公开协议、API 与跨实现互操作；辉夜计划对外接口与 ROS 2 包质量声明中常需此类测试。

## Golden File Test

**中文名称：** 黄金文件测试

将输出与已知正确快照比较的测试。适用于检测生成输出、序列化或 API 响应中的非预期变更；Agent 提示词渲染与结构化输出验证中常用。

## Performance Test

**中文名称：** 性能测试

测量延迟、吞吐量、资源消耗及不同负载下退化行为的测试。对 Agent 推理延迟、向量检索与具身控制循环时序等辉夜计划关键路径尤为重要。

## Load Test

**中文名称：** 负载测试

模拟预期生产流量规模的性能测试，验证系统在正常负载下无性能退化。用于 QA-L3 服务发布前的容量与 SLO 验证。

## Stress Test

**中文名称：** 压力测试

将系统推至预期极限之外的性能测试，识别崩溃点、退化行为与恢复特性。对评估 Error Budget 与 Incident 响应策略具有参考价值。

## Smoke Test

**中文名称：** 冒烟测试

部署后立即运行的最小测试集，验证系统基本可用。快速捕获灾难性部署失败；辉夜计划每次发布与 Preview Environment 创建后均须执行 Smoke Test。

## SLO (Service Level Objective)

**中文名称：** 服务级别目标（SLO）

以服务可靠性为目标的量化指标，以百分比或阈值表达（如「99.9% 的请求在 200ms 内完成」）。定义服务「足够可靠」的含义，是 Error Budget 与发布决策的基础。

## Error Budget

**中文名称：** 错误预算

由 SLO 推导出的允许不可靠额度。若 SLO 为 99.9%，错误预算为 0.1%。预算耗尽时，可靠性改进优先于新功能；辉夜计划通过 Error Budget Policy 规范此权衡。
