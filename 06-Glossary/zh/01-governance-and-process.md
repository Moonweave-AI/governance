# 治理与流程

本文档定义 Moonweave/辉夜计划（Kaguya Project）治理文档中使用的组织治理、决策制定与项目管理相关术语，帮助团队成员在跨领域协作时保持一致的语义与流程理解。

---

## Owner

**中文名称：** 负责人

对某项资产、系统、组件或文档负有长期责任的个人或团队。负责人确保其正确性、可维护性与延续性；与 DRI 不同，负责人的职责超越单次任务或里程碑，贯穿整个生命周期。辉夜计划中，每个 Area 与关键文档均须明确指定 Owner。

## DRI (Directly Responsible Individual)

**中文名称：** 直接责任人

推动特定任务、决策或倡议直至完成的唯一负责人。Owner 承担长期管养职责，而 DRI 则对将工作推进至下一门禁或里程碑负直接责任。每项进行中的工作项必须指定一名 DRI。

## Maintainer

**中文名称：** 维护者

拥有合并权限、受信任的贡献者，负责仓库或领域内的代码审查质量、架构一致性与社区健康。维护者是辉夜计划代码与文档质量的重要守门人。

## Contributor

**中文名称：** 贡献者

向项目提交变更（代码、文档、研究或其他产出物）的任何人。贡献者通常无合并权限，通过 Issue、PR 与讨论参与协作。

## Area

**中文名称：** 领域

项目内的责任域（如 Agent、Infra、Frontend、Backend、Data、Embodiment、Research）。每个 Area 均指定 Owner 与 Maintainer，便于职责划分与 CODEOWNERS 配置。

## CODEOWNERS

**中文名称：** CODEOWNERS（代码所有者机制）

GitHub 机制：当 PR 修改匹配指定路径模式的文件时，自动向指定个人或团队请求审查。辉夜计划借此确保关键路径变更由对应领域负责人审阅。

## RFC (Request for Comments)

**中文名称：** 征求意见稿

针对影响架构、公开 API、安全边界、跨仓库契约或长期维护的重大变更的正式提案文档。RFC 须经历结构化讨论、审查与共识形成后方可被接受；是辉夜计划重大决策的核心载体。

## ADR (Architecture Decision Record)

**中文名称：** 架构决策记录

记录已做出的架构决策的文档，包含背景、备选方案、理由与后果。ADR 为只增不改的日志——已接受的记录不得重写；被新决策取代时须新建 ADR。辉夜计划以此保留可追溯的架构演进历史。

## FCP (Final Comment Period)

**中文名称：** 最终评议期

RFC 被正式接受或拒绝前的限时窗口（通常 7–14 天），供所有利益相关方最后一次提出异议或关切。FCP 是辉夜计划共识流程中的关键缓冲环节。

## Rough Consensus

**中文名称：** 粗略共识

源自 IETF 的决策模型：目标并非全体一致，而是充分回应所有实质性技术异议。当剩余分歧已被听取、考量并记录后，即可推进决策——而非等待人人满意。辉夜计划对 RFC 等重大变更采用此模型。

## Lazy Consensus

**中文名称：** 惰性共识

轻量决策模型：若在规定的等待期内无人提出异议，则提案视为被接受。适用于低风险、可逆的变更，以提升日常治理效率。

## Bus Factor

**中文名称：** 巴士系数

在项目或系统变得无法维护之前，至少需要多少人同时不可用。巴士系数为 1 意味着关键知识仅存在于一人脑中——对长期系统而言是不可接受的风险。辉夜计划通过文档化、结对与 Area 多人负责制降低巴士系数。

## Gate

**中文名称：** 门禁

项目生命周期中的检查点，须满足特定标准后方可进入下一阶段。门禁防止不成熟或验证不足的工作过早推进；与成熟度等级 M0–M9 及质量门禁 Q0–Q5 紧密关联。

## Milestone

**中文名称：** 里程碑

标记一组交付物完成或特定项目目标达成的重要检查点，通常与时间线绑定。里程碑与 Gate 配合，用于跟踪辉夜计划各阶段的进展与交付。

## Moonweave Maturity (M0–M9)

**中文名称：** Moonweave 成熟度等级（M0–M9）

辉夜计划内部对工作项成熟度的分级尺度：自 M0（构想）经 M5（生产）至 M9（退役）。每一级对文档、测试、归属与运维就绪度有明确要求，是 Gate 与发布决策的依据。

## Security Level (S0–S5 + Blocked)

**中文名称：** 安全等级（S0–S5 + Blocked）

辉夜计划内部风险分类体系。S0 为最低风险（文档类）；S5 为最高运维风险（具物理后果的具身系统）。"Blocked" 表示在合规问题解决前工作不得继续。S3 及以上系统须完成威胁建模等额外安全要求。

## Stop-Ship

**中文名称：** 发布阻断

阻止任何发布或部署继续进行的条件。由未解决的安全漏洞、数据泄露、安全故障或超出可接受阈值的合规违规触发。辉夜计划在 Stop-Ship 状态下冻结发布直至风险消除。

## Postmortem

**中文名称：** 事后复盘

重大事件后进行的结构化回顾，聚焦理解发生了什么、为何发生，以及将实施哪些系统性改进以防复发。事后复盘秉持无责原则——针对流程与系统，而非个人。辉夜计划对 P0–P2 级事故要求撰写 Postmortem。
