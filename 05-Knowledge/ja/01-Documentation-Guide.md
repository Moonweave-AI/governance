# ドキュメント作成およびナレッジ資産管理規範

> 本文書は、輝夜計画におけるドキュメント、意思決定記録、研究記録、データ記録、モデル記録、運用記録の作成、保守、レビュー、バージョン管理、失効、およびアーカイブのルールを定義します。ドキュメントはコード完成後の付属物ではなく、システムを理解可能、保守可能、再現可能、監査可能、引き継ぎ可能にするインフラストラクチャです。エンジニアリング成果は、そのナレッジ記録が後任者が理解、検証、引き継ぎに十分である場合にのみ、真に完成したと言えます。

本文書は以下に代わるものではありません：

- `../../03-Collaboration/ja/03-RFC-Process.md`：重大な提案の正式な意思決定プロセス；
- `../../04-Engineering/ja/01-Workflow.md`：エンジニアリング実装ワークフロー；
- `../../04-Engineering/ja/02-Quality-Assurance.md`：品質エビデンス基準；
- `../../01-Foundation/ja/02-Security-Ethics.md`：セキュリティ、プライバシー、コンプライアンス、倫理的境界。

本文書は、上記ドキュメントが**扱わないナレッジ管理の問題**をカバーします：どのナレッジを記録しなければならないか？どこに記録するか？誰が保守を担当するか？どのように書けば読みやすく、レビュー可能で、再現可能で、引き継ぎ可能か？ドキュメントはいつ失効、廃止、または置き換えられるか？

---

## 1. 目的と適用範囲

本文書は、輝夜計画のナレッジ資産管理体系を定義します。

輝夜計画は単一のコードベースではなく、長期にわたって進化する AI Agent、Infra、研究、および具身システムです。多くのリスクは「コードがない」ことではなく、「なぜこのように設計したかを知っているのが特定の一人だけ」「あるチャットログにだけ言及があった」「ある会議で口頭で言っただけ」といった状況から生じます。本文書の目標は、このような暗黙知を追跡可能、レビュー可能、引き継ぎ可能な正式な資産に変えることです。

輝夜計画のすべての正式なリポジトリ、ドキュメントサイト、内部ナレッジベースにおいて適用されます：リポジトリ README、ユーザーおよび開発者向けドキュメント、API / プロトコル / Schema リファレンス、RFC / ADR / Engineering Brief / Design Doc、実験ログ / 論文 Review / 再現記録、Dataset Card / Model Card / Evaluation Report、Runbook / Incident Report / Postmortem / Release Notes。

ドキュメント体系全体は、Diátaxis の四分法（Tutorial / How-to / Reference / Explanation がそれぞれ異なる読者ニーズに対応）、Write the Docs の Docs as Code 理念（ドキュメントがコードと同じ Issue、Git、Review、自動化ワークフローを使用）、および GitLab がドキュメントをプロダクトの SSoT（Single Source of Truth）として扱う実践を取り入れています。

## 2. ドキュメント原則

8 条、ナレッジ資産管理を専門的に規定します：

1. **ドキュメントはナレッジ資産であり、付属説明ではない** — 他者に理解、保守、再現、レビュー、引き継ぎを必要とするナレッジは、すべてナレッジ資産として扱うべきです。ナレッジ資産には SSoT、Owner、ステータス、ライフサイクルが必要です。輝夜計画にとって、ナレッジの腐敗は「ドキュメントが追いついていない」ことではなく、「システムが理解不能、引き継ぎ不能になる」ことです。

2. **SSoT（Single Source of Truth）** — 各重要なナレッジポイントには、権威ある SSoT は一つだけ存在できます。他の場所では参照、要約、インデックス化は可能ですが、矛盾する事実を形成してはなりません。Feishu、Notion、PR コメント、正式ドキュメントの説明が一致しない場合、正式ドキュメント（または Git 内の対応する記録）が優先されます。

3. **Docs as Code** — 正式ドキュメントはデフォルトで Git 管理、PR による変更、Review の受け入れ、自動チェック、履歴追跡が可能です。ユーザー、インターフェース、デプロイ、保守方法に影響するコード変更には、ドキュメントの同期更新が必須です。重要なナレッジはチャットログにのみ存在してはなりません。

4. **タイプを選んでから内容を書く** — チュートリアル、操作手順、リファレンス資料、背景説明を同一ドキュメントに混在させないでください。執筆前に、読者が学習中（Tutorial）か、タスク完了中（How-to）か、事実照会中（Reference）か、背景理解中（Explanation）かを必ず判断してください。

5. **著者の経験ではなく、読者のタスクに向ける** — ドキュメントは、著者がシステムを実装した順序ではなく、読者が完了すべきタスクと理解すべき問題から始めるべきです。読者は開発日記に興味がありません — 「今何をすべきか」「なぜこうなっているか」を知りたいのです。

6. **重要な結論は必ず追跡可能であること** — なぜ特定のアーキテクチャを採用したか、なぜインターフェースを廃止したか、なぜモデル評価が合格したか、なぜ実験が失敗したか、なぜセキュリティ例外が受け入れられたか — これらの結論は Issue / PR / RFC / ADR / 実験ログ / 評価レポートまたはインシデント Postmortem にリンクしなければなりません。

7. **ドキュメントにはステータスと有効期間がある** — ドキュメントは永遠に正しいわけではありません。各正式ドキュメントは現在のステータスを表現できなければなりません：Draft / Active / Deprecated / Superseded / Archived。ステータスのないドキュメントは信頼できません。

8. **研究ドキュメントは再現を支援しなければならない** — 実験ログ、データ記録、モデル記録は、原作者が去った後でも、他者が実験目的、環境、入力、設定、結果、結論を理解できる十分な情報を保持しなければなりません。

## 3. ドキュメントタイプと SSoT

| ドキュメントタイプ | 目的 | 権威ある SSoT | 必須となるタイミング |
|-------------------|------|--------------|---------------------|
| README | 読者がプロジェクトの内容と開始方法を素早く判断できるようにする | 各リポジトリのルートディレクトリ | すべての正式リポジトリ |
| Tutorial | 新人を最初の成功体験まで導く | `docs/` またはドキュメントサイト | 外部ユーザーまたは新人向けの場合 |
| How-to | ユーザーが特定のタスクを完了するのを支援する | `docs/` / Runbook | 反復的な操作ニーズがある場合 |
| Reference | API、CLI、設定、Schema を正確に記述する | 生成ドキュメント + ソース仕様 | 公共インターフェースが存在する場合 |
| Explanation | 概念、背景、設計上のトレードオフを説明する | ドキュメントサイト / repo docs | 概念が複雑またはチーム横断の場合 |
| RFC | 重大な変更の提案、論証、意思決定 | `05-Knowledge/rfc/` | 重大な変更の前 |
| ADR | 既に下されたアーキテクチャ決定 | `05-Knowledge/adr/` | アーキテクチャ的決定の後 |
| Research Log | 実験、再現、ネガティブ結果 | `05-Knowledge/research-logs/` | 正式な研究または再利用可能な実験 |
| Dataset Card | データセットの出所、構成、ライセンス、用途、制限 | データリポジトリ / research logs | データセットが再利用、学習、評価される場合 |
| Model Card | モデルの用途、学習、評価、制限 | モデルリポジトリ / model registry | モデルが公開またはデプロイされる場合 |
| Runbook | 運用操作と障害対応 | repo docs / internal wiki | サービスがデプロイされている場合 |
| Release Notes | ユーザーにリリース変更を説明する | GitHub Release / changelog | 各正式リリース時 |
| Postmortem | インシデントの事実、影響、根本原因、アクション項目 | incident archive | P0/P1 または高リスクインシデントの後 |

コアルール：**一つの事実には一つの権威あるソースしか存在できません。** Notion に操作ガイドを書いたが、リポジトリ README に異なるバージョンがある場合、読者は誰を信じればよいかわからなくなります — これが SSoT 衝突です。

## 4. ドキュメントメタデータ

すべての正式ナレッジドキュメントは、ファイルヘッダーに最低限のメタデータを維持し、自動化ツールと Review がドキュメントステータスを判断できるようにするべきです：

**必須フィールド：**

- `owner` — このドキュメントの正確性を維持する責任者；
- `status` — Draft / Active / Deprecated / Superseded / Archived；
- `updated` — 最後の実質的更新日；
- `last_reviewed` — 最後に依然として正しいことを確認した日；
- `visibility` — public / internal / restricted；
- `type` — ドキュメントタイプ。

**推奨フィールド：**

- `audience` — 対象読者；
- `area` — 所属 Area（Agent / Infra / Frontend / Backend / Data / Embodiment / Research）；
- `review_cycle` — 再レビュー周期；
- `related_rfcs` / `related_adrs` / `related_issues` — 関連する意思決定と作業項目；
- `supersedes` / `superseded_by` — 置き換え関係；
- `sensitivity` — セキュリティ / プライバシー / IP 機密コンテンツを含むか。

メタデータ形式は YAML front matter またはドキュメント冒頭の構造化形式を採用でき、具体的な形式はリポジトリの規約によります。

## 5. README 規範

README は各リポジトリの入口です — 宣伝ページとして書くべきではなく、完全なマニュアルとしても書くべきではありません。GitHub Open Source Guides は、README が以下に答えることを推奨しています：プロジェクトが何をするか、なぜ有用か、どう始めるか、どこでヘルプを得るか。

### 5.1 README 推奨構造

適格な README には以下を含めるべきです：

- **プロジェクト名と一行での位置づけ** — 読者が 5 秒で関連性を判断できる；
- **ステータス** — Experimental / Active / Production / Deprecated / Archived；
- **プロジェクトが何をするか** — どの問題を解決するか；
- **輝夜計画における位置づけ** — 他のリポジトリおよびシステムとの関係；
- **Quick start** — 最短の実行可能パス；
- **コア概念** — 5 つ以内の重要用語；
- **使用方法** — 基本的な使用例；
- **開発** — ローカル開発、テスト、ビルドコマンド；
- **ドキュメントリンク** — 完全なドキュメント、API Reference、RFC / ADR へのリンク；
- **セキュリティ** — セキュリティ問題の報告方法；
- **貢献** — 参加方法；
- **ライセンス** — ライセンス；
- **Owner** — 責任者と連絡先。

### 5.2 README 禁忌

- スローガンのみで実行方法がない；
- インストール手順が失効しているが誰も更新しない；
- 現在のステータスの説明がない；
- Owner がない；
- 完全なアーキテクチャ設計ドキュメントを README に詰め込む；
- README が実際の CLI / API 動作と一致しない。

## 6. Tutorial / How-to / Reference / Explanation

本節は Diátaxis 四分法を上位分類として採用し、輝夜計画のシナリオに合わせて調整しています。

### 6.1 Tutorial

**目的**：読者を最初の成功体験まで導く。読者は学習中であり、仕事を完了しているわけではない。

適用シナリオ：Agent の初回実行、フロントエンド/バックエンドの初回起動、評価の初回実行、シミュレーション環境の初回使用。

執筆ルール：線形展開；各ステップは実行可能；結果の検証が必須；読者が内部アーキテクチャを理解していると仮定しない；大量の理論を展開しない — 背景は Explanation にリンク。

### 6.2 How-to

**目的**：基礎知識を持つユーザーが特定のタスクを完了するのを支援する。読者は何をしたいかを知っており、操作ガイダンスが必要。

適用シナリオ：バージョンのリリース方法、API endpoint の追加方法、モデル評価の実行方法、flaky test の対処方法、データベースマイグレーションの実行方法。

執筆ルール：タイトルは「〜の方法」形式；一つのタスクのみを解決；条件は指示の前に配置；指示は番号付きリスト；過度な背景説明をしない。

### 6.3 Reference

**目的**：正確、完全、検索可能な事実を提供する。読者は特定のパラメータ、フィールド、動作を照会している。

適用シナリオ：API Reference、CLI Reference、設定項目、Schema、ステートマシン、エラーコード、Agent Tool Interface。

執筆ルール：中立的、完全、検索可能；チュートリアル叙述を混在させない；自動生成可能なものはソース仕様からの生成を優先；公共インターフェース Reference にはバージョンと互換性を明記。

Microsoft のリファレンスドキュメントガイドは、リファレンスドキュメントが標準化された構造と一貫した表現により、開発者が必要な情報を素早く見つけられるよう支援すべきであると強調しています。

### 6.4 Explanation

**目的**：背景、原理、設計上のトレードオフ、概念モデルを説明する。読者は理解を求めている。

適用シナリオ：状態永続化の設計思想、ドメイン契約統一の理由、Agent Memory アーキテクチャ、具身安全境界の原理、評価体系の設計。

執筆ルール：叙述と背景を含めてよい；操作手順を装わない；RFC / ADR / Reference にリンク；正式な意思決定記録に代わってはならない。

## 7. API、プロトコル、生成ドキュメント

輝夜計画はフロントエンド、バックエンド、Agent、Infra、データ、具身実行を横断します。公共契約のドキュメントは強い制約が必要です。

### 7.1 基本要件

すべての公共 API、Schema、Protocol、Event、State Machine、Agent Tool Interface、データ形式には、権威ある仕様ドキュメントが必要です。最低限以下を含みます：バージョン、Owner、ステータス、互換性ポリシー、フィールド説明、エラー意味論、例、移行説明。

### 7.2 生成ドキュメントルール

- ソース仕様（OpenAPI / Proto / GraphQL SDL）から自動生成可能な Reference は、手書きの重複版を作成しない；
- 手書きドキュメントは「なぜ」「どう使うか」を説明し、生成ドキュメントは「何であるか」を記述する；
- 公共契約の変更は RFC / ADR / PR にリンクする；
- Breaking change には移行ドキュメントとバージョン説明が必須。

Kubernetes ドキュメント体系は「手書き説明 + 自動生成リファレンス」モデルを採用しています：API Reference、kubectl Reference などはソースコードから自動生成され、概念とタスクガイドは人間が執筆します。輝夜計画も同じ役割分担を採用すべきです。

## 8. アーキテクチャドキュメント、RFC、ADR

### 8.1 RFC

RFC は重大な変更の提案背景、問題、方案、代替案、リスク、議論、最終決定を記録します。回答する問い：**こうすべきか？なぜか？**

RFC の完全なプロセスとテンプレートは `../../03-Collaboration/ja/03-RFC-Process.md` で定義されています。本文書はドキュメント管理面のルールのみを強調します：RFC は履歴記録であり、Accepted / Rejected / Superseded の RFC は遡及的に大幅修正してはなりません。後続の行動は ADR、標準ドキュメント、または実装 Issue で維持します。

### 8.2 ADR

ADR は既に下されたアーキテクチャ決定を記録します：当時のコンテキスト、決定内容、除外された方案、理由、結果。回答する問い：**何を決定したか？どのような結果をもたらすか？**

ADR は append-only log として維持すべきです — 受け入れられた記録は遡及的に書き換えない；決定が変わった場合は新しい ADR を書き、元の記録にリンクする。Microsoft Azure Well-Architected Framework は、ADR はシステム構造、重要な品質属性、または逆転が困難な決定に影響するもののみを記録すべきと提案しています。

ADR 推奨構造：Status（Proposed / Accepted / Deprecated / Superseded）→ Context → Decision → Consequences → Alternatives Considered → Related RFCs → Superseded By。

### 8.3 Engineering Brief と Design Doc

Engineering Brief は `../../04-Engineering/ja/01-Workflow.md` §6 で定義されています。Design Doc は RFC 受け入れ後、実装開始前の詳細な技術設計です。両方ともナレッジ資産であり、Owner とステータスを持ち、変更は PR Review を経由します。

## 9. Research Logs、Paper Review、実験記録

`05-Knowledge/research-logs/` は輝夜計画において非常に重要なナレッジディレクトリです。プロジェクトは Agent 研究、AI Infra 探索、具身知能実験を含みます — ここには「成功結果」だけでなく「プロセス」と「失敗」も保存しなければなりません。

### 9.1 実験ログ

各正式実験または再利用可能な探索的作業には以下を記録します：研究問題、仮説、動機、実験環境（コードバージョン / データバージョン / モデルバージョン / 設定 / ハードウェア / 乱数シード）、方法、評価指標、生の結果、分析、ネガティブ結果、限界、再現方法、結論。

コアルール：

- 失敗した実験も記録必須 — ネガティブ結果は重複した浪費を防ぐ；
- 結果はコード、設定、データ、環境バージョンにリンクする；
- 再現できない場合は理由を説明する；
- 実験結論は適用範囲を誇張してはならない；
- 実験ステータスを明記：進行中 / 完了 / アーカイブ / 無効。

The Turing Way は再現性を、データとコードが利用可能で、分析が完全に再実行可能であることと定義しています。輝夜計画の研究ログはこれを最低基準とすべきです。

### 9.2 Paper Review

輝夜計画に関連する論文のレビュー記録：引用情報、要約、問題、方法、エビデンス、長所と短所、仮定、輝夜計画への意義、再現性評価、リスク評価、後続アクション（再現 / 実験 / RFC / エンジニアリング化が必要か）。

### 9.3 データセット記録

正式に使用される各データセットには完全な記録を維持：識別情報（名称 / バージョン / Owner / 出所 / ライセンス / ストレージ / アクセス制御）、動機、構成、収集プロセス、前処理、許可用途、禁止用途、プライバシーと認可の問題、バイアスと限界、品質チェック結果、バージョン履歴。

Datasheets for Datasets は、データセットに動機、構成、収集プロセス、推奨用途を説明するドキュメントを添付すべきと提案しています。FAIR 原則は、デジタル資産が Findable、Accessible、Interoperable、Reusable であることを要求します。輝夜計画で正式に再利用または公開されるデータセットは、両方の要件を満たすべきです。

## 10. Dataset Card、Model Card、AI 資産記録

### 10.1 Dataset Card

Dataset Card はデータ利用者向けで、データセットが自分の用途に適しているかを判断するのを支援します。最低内容：名称とバージョン、出所とライセンス、規模と形式、データ構成の説明、前処理の説明、推奨用途と禁止用途、バイアスと限界、プライバシーと機密情報の声明、品質チェック結果、引用方法、関連実験/モデル。

Hugging Face Dataset Cards の実践は形式参考として使用できますが、輝夜計画では出所コンプライアンスを追加で強調すべきです（`../../01-Foundation/ja/02-Security-Ethics.md` §4 資産と出所セキュリティに整合）。

### 10.2 Model Card

Model Card はモデル利用者とレビュアー向けです。最低内容：名称とバージョン、ベースモデル、学習データソース、学習パラメータ、評価結果と指標、適用シナリオと範囲外の使用、制限とバイアス、セキュリティ注意事項、推論コストとレイテンシ、既知の失敗モード、ロールバックまたは代替案、引用方法。

Model Cards for Model Reporting は、公開モデルに適用境界、評価プロセス、限界を説明するドキュメントを添付することを要求しています。輝夜計画の各正式モデルサービスは Model Card を維持すべきです。

### 10.3 Agent Behavior Record

輝夜計画固有のドキュメントタイプ：ツール呼び出し、長期記憶、ユーザーインタラクション、外部アクション能力を持つ Agent にはすべて行動記録が必要です。内容：Agent 識別情報とバージョン、能力範囲、ツールリストと権限境界、記憶戦略、ペルソナ/行動制約、評価結果、既知の失敗モード、セキュリティ緩和策、人間への引き継ぎ方法、監査ログ要件、バージョン履歴。

## 11. Runbook、Incident、Release、運用ドキュメント

### 11.1 Runbook

Runbook は保守者向けであり、一般ユーザーのチュートリアルではありません。本番サービスには Runbook が必須です。

内容は以下をカバー：システム用途、Owner とエスカレーションパス、システム概要と依存関係、監視 Dashboard リンク、アラート説明、一般的な操作手順、インシデント対応フロー、ロールバック方法、バックアップ復旧方法、既知の障害モード、関連リンク。

ルール：Runbook はインシデント中に使用可能であること（時間的プレッシャー下でも操作可能）；キーや認証情報を記載してはならない；定期的な演習またはレビューが必須。

### 11.2 Incident Report / Postmortem

P0/P1 または高リスクインシデントの後は Incident Report を必ず作成し、以下を記録：要約、影響範囲と期間、タイムライン、検知方法、根本原因、寄与要因、解決方法、うまくいった点、うまくいかなかった点、アクション項目（Owner / Due / Tracking Issue を含む）、後続ドキュメント更新。

Postmortem の目的は教訓を蓄積し再発を防ぐこと — 個人の責任追及ではありません。アクション項目は追跡可能でなければならず、「次回注意」で終わってはなりません。

### 11.3 Release Notes

Release Notes はユーザーと保守者向けであり、Git log のコピーではありません。以下を含む：重点変更の要約、Breaking change、新機能、修正、セキュリティ更新、移行説明、既知の問題、貢献者への謝辞。コードを読まないユーザーでも影響を理解できる表現にします。

## 12. 執筆スタイルと言語規範

### 12.1 基本スタイル

正確、簡潔、直接的、タスク指向。宣伝調、未定義の略語、抽象名詞の積み重ね、不確実な結論を事実として書くことを避けます。Google Developer Documentation Style Guide と Microsoft Writing Style Guide の共通の助言：能動態、二人称、明確な条件順序、記述的リンクを使用する。

### 12.2 中国語と英語のルール

輝夜計画のドキュメントは主に中国語です。英語用語は以下のルールに従います：

- 初出：中国語用語（English Term）または English Term（中国語の説明）；
- 以降は確立された呼称を優先し、同一リポジトリ内で同一概念に複数の訳名を使用しない；
- 固有名詞、プロトコル名、ツール名、API 名は原文のまま；
- 不確実な訳名は Glossary に登録し、各ドキュメントで独自に翻訳しない。

### 12.3 禁止表現

以下の表現は正式ドキュメントで避けるか置き換えるべきです：

- 「明らかに」「簡単に」「〜するだけ」「とても簡単」 — 読者のコストと環境差を隠す；
- 「周知の」「言うまでもない」 — 読者の知識を仮定する；
- 「絶対に安全」「完璧」「決して〜ない」 — リスクと限界の境界を隠す；
- 対応する品質エビデンスのない「本番利用可能」 — 未検証の断言はドキュメントではない。

## 13. Markdown、コード例、図表、メディア

### 13.1 Markdown 規範

- GitHub Flavored Markdown（GFM）を使用；
- 各ドキュメントに H1（タイトル）は一つのみ；
- 見出しレベルを飛ばさない（H2 から H4 に直接進まない）；
- 同一リポジトリ内のドキュメントは相対リンクで参照；
- コードブロックには言語を指定；
- コマンド例はコピー可能（無関係な shell prompt 記号を含まない）；
- テーブルは構造化比較用であり、長文本文用ではない；
- 画像には alt text が必須；
- 出所を追跡できないメディア資産をコミットしない。

### 13.2 コード例

コード例はドキュメントにおける実行可能な約束です。要件：

- 最小限で実行可能、著者のローカルパスや暗黙の環境に依存しない；
- 依存バージョンを明示；
- 実際のキーや認証情報を含まない；
- 期待される出力がある；
- 現在の API / CLI バージョンと一致；
- 重要な例は CI テスト（doc test）に組み込む。

### 13.3 図表

- 単純なフローチャートは Mermaid を使用（バージョン管理可能、diff 可能）；
- 複雑なアーキテクチャ図はソースファイル + エクスポート画像を保持（両方コミット）；
- UI スクリーンショットにバージョンと日付を注記（スクリーンショットは失効する）；
- 研究図表は生成スクリプトとデータバージョンにリンク。

図表にはソースファイルを特定できなければならない；図表内でプライバシー、キー、未公開情報を漏洩してはならない；アーキテクチャ図は関連 RFC / ADR にリンクする。

## 14. ドキュメントワークフローと Review

### 14.1 ドキュメント変更フロー

```text
Identify doc need
  ↓
Choose doc type
  ↓
Create draft (PR)
  ↓
Technical review
  ↓
Docs review (readability, accuracy, completeness)
  ↓
Security / privacy / AI / IP review (if applicable)
  ↓
Merge
  ↓
Publish / index
  ↓
Periodic review
  ↓
Update / deprecate / archive
```

### 14.2 ドキュメント同期更新が必須の場合

いずれかの条件を満たす場合、PR にはドキュメント更新または新規ドキュメントを含める必要があります：

- 公共 API / Schema / CLI / 設定の追加、変更、削除；
- デプロイ、環境変数、実行方法の変更；
- Agent 行動、ツール権限、記憶戦略の変更；
- モデル、データセット、評価方法の変更；
- セキュリティ、プライバシー、コンプライアンス境界の変更；
- 具身制御、シミュレーション、アクチュエータ行動の変更；
- 新バージョンのリリース；
- 機能の廃止またはアーカイブ；
- 使用方法または保守方法に影響するインシデントの発生。

### 14.3 ドキュメント Review 要件

| ドキュメントタイプ | 必須 Review |
|-------------------|-------------|
| README | Repo Owner |
| Tutorial / How-to | 技術 Reviewer + 対象読者視点 |
| Reference | API / Schema Owner |
| RFC | `../../03-Collaboration/ja/03-RFC-Process.md` で定義された Reviewer |
| ADR | アーキテクチャ Owner / Maintainer |
| Research Log | Research Reviewer |
| Dataset Card | Data Owner + License / Privacy Reviewer |
| Model Card | AI Owner + Eval Reviewer |
| Runbook | Service Owner + Infra / SRE Reviewer |
| セキュリティ機密ドキュメント | Security Reviewer |

## 15. ドキュメントバージョン、失効、廃止、アーカイブ

### 15.1 ドキュメントステータス定義

| ステータス | 意味 |
|-----------|------|
| **Draft** | 草稿、SSoT として使用不可 |
| **Active** | 現在有効、権威ある事実 |
| **Deprecated** | 閲覧可能だが、新規作業の根拠としては非推奨 |
| **Superseded** | 新ドキュメントに置き換え済み、代替ドキュメントにリンク |
| **Archived** | 履歴記録、監査と追跡価値のみ保持 |

### 15.2 再レビュー周期

| ドキュメントタイプ | デフォルト再レビュー周期 |
|-------------------|------------------------|
| 原則 / ガバナンス / エンジニアリング標準 | 6–12 か月 |
| README | 3–6 か月 |
| API / Reference | 各 release 時 |
| Runbook | 関連インシデントの都度 + 四半期ごと |
| RFC | 履歴を変更せず、superseding RFC のみ追加 |
| ADR | 履歴を変更せず、superseding ADR のみ追加 |
| Research Log | 実験終了後にアーカイブ |
| Dataset / Model Card | データまたはモデルバージョン変更の都度 |
| セキュリティドキュメント | 四半期ごとまたは重大リスク変化後 |

### 15.3 バージョン管理ドキュメント

ユーザーが特定のソフトウェアバージョンに応じて異なる説明を読む必要がある場合にのみ、バージョン管理ドキュメントを有効化します。Kubernetes は現在バージョンと前 4 バージョンのドキュメントのみを維持しています — この限定ウィンドウは「すべての履歴バージョンを永続的に維持する」より持続可能です。輝夜計画は正式な公共 API が安定してからドキュメントバージョン管理を導入し、早期の保守負担増加を避けるべきです。

### 15.4 ドキュメント廃止フロー

ドキュメント廃止は削除ではありません。フロー：Deprecated または Superseded とマーク → ドキュメントヘッダーに代替パスを追加 → 関連インデックスを更新 → アーカイブ日を設定 → アーカイブ後、履歴ディレクトリに移動または Archived とマーク。アーカイブ済みドキュメントはバージョン管理に保持し、監査と追跡を支援します。

## 16. ドキュメント品質チェックと自動化

ドキュメントも CI に組み込むべきです。

### 16.1 自動化チェック基準

正式ドキュメントリポジトリの CI には以下を含める：

- Markdown lint（見出しレベル、形式の一貫性）；
- リンクチェック（broken links、broken anchors）；
- スペルチェック；
- Prose lint（Vale または同等ツール、スタイルルールチェック）；
- Front matter 検証（Owner / status / updated の存在確認）；
- コードブロック構文検証；
- 禁止パターンスキャン（secret / token / ハードコード認証情報）；
- 生成ドキュメント一貫性チェック（生成 Reference がソース仕様と同期しているか）。

Vale は prose 向け linter で、カスタムスタイルルールをサポートします；markdownlint は Markdown 静的解析ツールです。両者を組み合わせることで、ドキュメントの構造と言語品質をカバーできます。

### 16.2 Kaguya Doc Steward

輝夜計画はドキュメント Agent をデプロイしてナレッジ管理を支援できます：Owner 欠如または review date 失効のドキュメントチェック、broken links チェック、README に status / Owner / Quick start が欠如していないかチェック、RFC 承認後に対応する Implementation Issue があるかチェック、ADR に superseded リンクが欠如していないかチェック、Research Log に data / code / configuration / results が欠如していないかチェック、週次 Documentation Health Digest の生成。

> Kaguya Doc Steward はリマインド、要約、草稿 PR の作成が可能です；事実的結論を自動変更してはなりません；Owner によるドキュメント有効性確認に代わってはなりません；制限付きドキュメントの要約を公開チャンネルにプッシュしてはなりません。

## 17. AI 支援執筆規範

輝夜計画は AI 支援ドキュメント作業を許可しますが、ドキュメントは常に人間が責任を負います。

**許可：**

- 構造とアウトラインの起草；
- 文の書き換えと推敲；
- 会議記録の要約；
- RFC / ADR からの要約起草；
- 用語一貫性チェック；
- FAQ または Glossary エントリの初版生成。

**禁止：**

- 未検証の AI 出力を事実として公開；
- AI を使用して引用、実験結果、API 動作、履歴上の決定を捏造；
- 未公開コード、キー、ユーザーデータ、研究データを外部 AI サービスに入力；
- AI を使用して RFC、ADR、Research Log、セキュリティドキュメントを自動承認；
- 著者が説明できない技術ドキュメントを AI で生成。

> AI-assisted documentation remains human-owned documentation. Author は内容に対して完全な責任を負います。

### Definition of Done for Documentation

正式ドキュメントは、以下の条件をすべて満たした場合にのみ完成とみなされます：

- ドキュメントタイプが正しく選択されている；
- Owner がある；
- 明確なステータスがある；
- 対象読者が定義されている；

- 必要な Review が完了している；
- インデックスまたはナビゲーションに追加されている；
- 次回再レビュー時刻が設定されている。

## 19. アンチパターン

以下の行為はドキュメントとナレッジ管理のアンチパターンです：

1. コードのみで README がない；
2. README にプロジェクトスローガンのみで Quick start と現在のステータスがない；
3. ドキュメントの事実が Feishu、Notion、PR コメント、個人の記憶に散在し、権威ある SSoT がない；
4. チュートリアル、リファレンス、説明、意思決定記録が一つのドキュメントに混在；
5. PR がインターフェースを変更したが Reference と使用ドキュメントを更新しない；
6. RFC 承認後に ADR、Implementation Issue、標準ドキュメント更新がない；
7. ADR を遡及的に書き換え、superseding ADR を追加しない；
8. 実験が成功結果のみを記録し、設定、データバージョン、失敗を記録しない；
9. データセットに出所、ライセンス、用途、制限の説明がない；
10. モデル公開に Model Card と Eval Report がない；
11. Runbook がインシデント後に口頭でのみ伝授される；
12. 図表にソースファイルがなく、更新と追跡ができない；
13. 例示コードが実行不可だが説明もない；
14. ドキュメントに Owner がなく、正確性の責任者がいない；
15. Deprecated コンテンツに代替パスの案内がない；
16. ドキュメントが失効しているが依然として SSoT として使用されている；
17. AI 生成ドキュメントが人間による検証を経ていない；
18. 「完全性」のために長文を書くが、読者がそこからいかなるタスクも完了できない。

## 20. 改訂

本文書は公開 RFC による改訂のみ可能です。改訂には、ナレッジ管理実践が過重または不足であることが証明されたか、新タイプの資産にカバレッジが欠如しているか、あるルールが実践においてナレッジ蓄積を促進せず阻害しているかを明記する必要があります。`../../01-Foundation/ja/01-Principles.md` の「衝突と改訂」と一致：本文書がセキュリティレビューまたは組織権限と衝突する場合、対応する専門ドキュメントが優先；底線原則が優先。旧版はバージョン管理に保存され、いつでも参照可能。

本文書のハードコアは 5 条のみ：

1. すべての正式ナレッジには SSoT が必要。
2. すべての正式ドキュメントには Owner とステータスが必要。
3. すべての重要なエンジニアリングと研究結論は追跡可能であること。
4. ドキュメント変更は Review と自動化チェックに組み込むこと。
5. 失効したナレッジはマーク、置き換え、またはアーカイブすること。

この 5 条を守れば、`05-Knowledge/` は単なる資料フォルダではなく、輝夜計画の長期組織記憶システムとなります：なぜ行ったか、どう行ったか、何を試したか、何が失敗したか、何を決定したか、後任者がどう引き継ぎを続けるかを記録します。
